# 角色菜单权限规划（基于实际工作岗位）

> 适用系统：AssetHub 医学装备整体运维管理系统
> 规划依据：实际医院医学装备科（处）岗位分工
> 数据源：`backend` 中的 `menu_definitions`（菜单定义表）与 `role_menu_permissions`（角色菜单权限表）——即前端 `Layout` 通过 `/roles-permissions/user/menus` 实际渲染的侧边栏来源。
> 配套文件：`role-menu-permissions.json`（可直接用于初始化 `role_menu_permissions` 的机器可读配置）

---

## 1. 设计目标与原则

不同角色进入系统后，应**只看到并完成自己职责范围内的功能**，做到：

1. **按岗位而非按人设权**：权限挂在"角色"上，人通过 `user_tenant_roles` 关联角色，便于轮岗与审计。
2. **最小够用**：业务专员只进自己业务域；管理类（用户/角色/备份/连接）集中给系统管理员。
3. **查看 / 经办 / 管理 三级分离**：
   - 普通用户、科室管理员：**查看 + 发起申请**（如维修申请、调配申请、不良事件上报）。
   - 业务管理员（验收/维修/计量/质控/调配/盘点）：**经办 + 审核 + 配置**本业务域。
   - 系统/超级管理员：**系统级配置**，不掺杂具体业务数据操作。
4. **菜单可见性 ≠ 数据范围**：菜单决定"能看到哪个入口"，`role_data_scopes` 决定"能看到哪些数据"（全院 / 本科室 / 本人）。两者配合 `restrictToOwnDepartment` 中间件生效。

## 4. 数据范围（role_data_scopes）

菜单决定"看不看得见入口"，数据范围决定"看得见多少数据"。建议配置：

| 角色 | 数据范围 `data_scope` | 说明 |
|---|---|---|
| `super_admin` / `system_admin` | `all` | 全机构 |
| `asset_admin` | `all` | 全院资产台账 |
| `acceptance_admin` / `metrology_admin` / `quality_admin` / `transfer_admin` / `inventory_admin` / `maintenance_admin` | `all` | 全院本业务域；多院区可改 `department` + `custom_department_codes` |
| `department_admin` | `department` | 限本科室（`managed_departments`） |
| `maintenance_engineer` | `department` | 限本院维修范围 |
| `user` | `own` / `department` | 本人或本科室 |

## 5. 现状问题与修正建议

规划落地前，需先处理以下系统现状问题（来自代码核查）：

1. **`maintenance_engineer` 未种子进 `roles` 表**
   该角色在 `roleMenuPermissions` 默认规则与前端 `BUSINESS_ADMIN_ROLES` 中均被引用，但 `create-roles-table.js` 的 `defaultRoles` 种子列表里**没有它**，导致新建用户无法选到该角色。
   → 在 `defaultRoles` 中补一条 `maintenance_engineer`（维修工程师），或明确并入 `maintenance_admin`。

2. **存在两套菜单树，且不一致**
   - 系统 A：`menu_definitions`（后端种子）→ 驱动前端 `Layout` 实际渲染。
   - 系统 B：`routes/menus.js` 的 `ALL_MENUS`（带 `module_id`、角色门控，菜单更细）。
   两者父级标签、挂接关系不一（如盘点：A 挂在 `/assets-parent` 下，B 独立为 `/inventory-parent`）。
   → **以系统 A（`menu_definitions` + `/roles-permissions/user/menus`）为唯一权威**，废弃或对齐系统 B，避免"配了菜单但前端不显示"。

3. **`/tenants`（企业管理）对 `system_admin` 处理不一致**
   种子里 `system_admin = 1`（全可见），但 `/roles-permissions/user/menus` 端点会额外剔除 `/tenants`。
   → 明确规则：**企业管理仅 `super_admin` 可见**；种子与端点保持一致。

4. **`adminOnlyMenuKeys` 仅对业务角色生效**
   `/dashboard-configs`、`/cloud-sync`、`/modules`、`/tenant-module-config` 对业务角色隐藏，但 `super_admin`/`system_admin` 仍可看。其中 `/database-connection`（连接管理）、`/cloud-sync`（云/IoT 同步）属高敏感操作。
   → 建议将 `/database-connection`、`/cloud-sync` 收紧为**仅 `super_admin`**；`/modules`、`/dashboard-configs` 保留给系统管理员。

5. **菜单可见性与数据范围需配套**
   仅收敛菜单不够，例如 `department_admin` 看得到"资产列表"但必须靠 `role_data_scopes=department` + `restrictToOwnDepartment` 才能真正只看到本科室数据。
