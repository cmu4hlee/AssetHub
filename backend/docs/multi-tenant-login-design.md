# 多企业版本用户登录逻辑设计

## 1. 概述

本文档设计了一个适合多企业版本的用户登录逻辑，基于现有的前后端分离架构，结合租户（企业）管理，提供安全、便捷的登录体验。

## 2. 核心设计原则

- **安全性优先**：严格的身份验证和授权机制
- **用户体验优化**：简化登录流程，提供清晰的错误提示
- **租户隔离**：确保不同企业用户数据的完全隔离
- **灵活性**：支持不同角色的登录需求
- **可扩展性**：方便未来功能扩展

## 3. 登录流程设计

### 3.1 完整登录流程

```
用户访问登录页面 → 输入企业编码 → 系统验证企业编码 → 显示用户名密码输入框 → 输入用户名密码 → 系统验证用户身份和权限 → 登录成功 → 跳转到相应页面
```

### 3.2 登录流程图

```
[用户] → [输入企业编码] → [系统验证企业编码] → [显示用户名密码框] → [输入用户名密码] → [系统验证身份] → [分配租户权限] → [生成令牌] → [登录成功]
     ↓                      ↓                                ↓                                ↓                                ↓                                ↓
[错误提示] ← [企业不存在/已停用] ← [用户名不存在/密码错误] ← [用户不在该企业] ← [用户已禁用] ← [租户已过期] ← [生成令牌失败]
```

## 4. 关键功能设计

### 4.1 企业编码验证机制

- **企业编码**：每个企业拥有唯一的企业编码（tenant_code）
- **验证时机**：登录流程的第一步
- **验证内容**：
  - 企业编码是否存在
  - 企业状态是否为"active"
  - 企业订阅是否过期
- **用户体验**：提供企业编码输入框，实时验证企业信息

### 4.2 多角色登录支持

#### 4.2.1 超级管理员（super_admin）

- 无需企业编码登录
- 可以管理所有企业和用户
- 登录后默认进入企业管理页面

#### 4.2.2 系统管理员（system_admin）

- 可以选择企业编码登录（适用于管理多个企业的情况）
- 也可以不输入企业编码，使用已有租户信息登录
- 登录后只能管理所属企业的资源

#### 4.2.3 普通用户（asset_admin/department_admin/user）

- 必须输入企业编码登录
- 只能访问所属企业的资源
- 登录后默认进入个人工作台

### 4.3 租户自动分配机制

- **新用户**：首次登录时自动分配到指定企业
- **现有用户**：验证是否属于指定企业
- **跨企业限制**：普通用户不允许跨企业登录

### 4.4 令牌设计优化

- **JWT令牌**：使用JSON Web Token进行身份验证
- **令牌内容**：包含用户ID、用户名、角色、部门、租户ID等信息
- **过期时间**：默认30天，可以配置
- **刷新机制**：支持令牌刷新，避免频繁登录

### 4.5 错误处理与提示

| 错误类型         | 错误信息                             | 处理建议                   |
| ---------------- | ------------------------------------ | -------------------------- |
| 企业编码不存在   | "企业编码不存在，请检查输入"         | 提示用户检查企业编码       |
| 企业已停用       | "该企业已被停用，请联系管理员"       | 提示用户联系超级管理员     |
| 企业订阅过期     | "该企业订阅已过期，请联系管理员续费" | 提示用户联系企业管理员     |
| 用户名不存在     | "用户名或密码错误"                   | 不暴露具体错误信息         |
| 密码错误         | "用户名或密码错误"                   | 不暴露具体错误信息         |
| 用户已禁用       | "用户已被禁用，请联系管理员"         | 提示用户联系企业管理员     |
| 用户不属于该企业 | "该用户不属于指定的企业"             | 提示用户输入正确的企业编码 |

## 5. 数据库设计

### 5.1 用户表（users）

| 字段名     | 类型         | 描述                                                               |
| ---------- | ------------ | ------------------------------------------------------------------ |
| id         | INT          | 用户ID                                                             |
| username   | VARCHAR(50)  | 用户名                                                             |
| password   | VARCHAR(255) | 密码（加密存储）                                                   |
| real_name  | VARCHAR(100) | 真实姓名                                                           |
| role       | VARCHAR(50)  | 角色（super_admin/system_admin/asset_admin/department_admin/user） |
| tenant_id  | INT          | 租户ID（关联到tenants表）                                          |
| status     | VARCHAR(20)  | 状态（active/inactive）                                            |
| created_at | TIMESTAMP    | 创建时间                                                           |
| updated_at | TIMESTAMP    | 更新时间                                                           |

### 5.2 租户表（tenants）

| 字段名                  | 类型         | 描述                       |
| ----------------------- | ------------ | -------------------------- |
| id                      | INT          | 租户ID                     |
| tenant_code             | VARCHAR(50)  | 企业编码（唯一）           |
| tenant_name             | VARCHAR(200) | 企业名称                   |
| contact_person          | VARCHAR(100) | 联系人                     |
| contact_phone           | VARCHAR(20)  | 联系电话                   |
| status                  | VARCHAR(20)  | 状态（active/inactive）    |
| subscription_type       | VARCHAR(50)  | 订阅类型（free/basic/pro） |
| subscription_start_date | DATE         | 订阅开始日期               |
| subscription_end_date   | DATE         | 订阅结束日期               |
| max_users               | INT          | 最大用户数                 |
| max_assets              | INT          | 最大资产数                 |
| created_at              | TIMESTAMP    | 创建时间                   |
| updated_at              | TIMESTAMP    | 更新时间                   |

## 6. API设计

### 6.1 验证企业编码

```
POST /api/tenants/verify
请求体：{ "tenant_code": "company001" }
响应：
{ "success": true, "data": { "tenant_id": 1, "tenant_name": "企业A", "status": "active" } }
或
{ "success": false, "message": "企业编码不存在" }
```

### 6.2 用户登录

```
POST /api/users/login
请求体：{ "username": "admin", "password": "123456", "tenant_code": "company001" }
响应：
{
  "success": true,
  "data": {
    "user": { "id": 1, "username": "admin", "role": "system_admin", "tenant_id": 1, "tenant_name": "企业A" },
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}
或
{ "success": false, "message": "用户名或密码错误" }
```

### 6.3 令牌刷新

```
POST /api/users/refresh-token
请求头：Authorization: Bearer <旧令牌>
响应：
{ "success": true, "data": { "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." } }
或
{ "success": false, "message": "令牌已过期" }
```

## 7. 前端实现建议

### 7.1 登录页面设计

- **第一步**：企业编码输入框
- **第二步**：用户名密码输入框（验证企业后显示）
- **记住企业**：提供记住企业编码的选项
- **错误提示**：实时显示验证结果和错误提示
- **响应式设计**：适配不同设备

### 7.2 登录状态管理

- 使用localStorage存储令牌和用户信息
- 定期检查令牌有效期
- 自动刷新即将过期的令牌
- 登录状态变更时通知相关组件

### 7.3 企业切换功能

- 系统管理员支持企业切换
- 提供企业切换下拉菜单
- 切换企业时重新验证权限

## 8. 安全机制

### 8.1 身份验证

- 使用bcryptjs加密存储密码
- 严格的密码复杂度要求
- 支持密码找回功能
- 登录失败次数限制（防止暴力破解）

### 8.2 授权控制

- 基于角色的访问控制（RBAC）
- 细粒度的权限管理
- 租户级别的数据隔离
- 动态权限验证

### 8.3 数据安全

- 企业数据完全隔离
- 敏感数据加密传输
- 定期数据备份
- 安全审计日志

## 9. 性能优化

- 使用缓存减少数据库查询
- 令牌验证优化
- 登录流程异步处理
- 前端资源按需加载

## 10. 测试建议

- 单元测试：验证各个登录组件的功能
- 集成测试：验证完整登录流程
- 安全测试：验证身份验证和授权机制
- 性能测试：验证登录过程的响应时间
- 兼容性测试：验证不同浏览器和设备的兼容性

## 11. 部署建议

- 前端使用CDN加速
- 后端服务负载均衡
- 数据库主从复制
- 定期安全更新

## 12. 总结

本设计提供了一个完整的多企业版本用户登录逻辑，结合了安全性、用户体验和灵活性。通过企业编码验证、多角色支持、租户隔离等机制，确保了系统的安全稳定运行，同时提供了良好的用户体验。

该设计基于现有的系统架构，易于实施和扩展，可以很好地满足多企业版本的登录需求。
