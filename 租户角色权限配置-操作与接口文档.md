# 租户角色权限配置 —— 操作说明与接口文档

> 适用模块：多租户用户角色权限管理系统（需求 ④：角色与权限可由租户管理员配置，不同租户间相互独立）
> 后端端口：`5183`　前端（Vite dev）端口：`13579`

---

## 1. 概述

本功能允许**各租户的系统管理员（system_admin）**在登录后，针对**当前所在租户**内的每个标准角色，独立配置三类权限：

| 配置项 | 含义 | 存储表 |
|--------|------|--------|
| 菜单可见性 | 该角色在当前租户下能看到哪些左侧菜单 | `tenant_role_menus` |
| 数据范围 | 该角色能访问的数据边界（全院 / 本科室 / 仅本人 / 自定义科室） | `tenant_role_data_scopes` |
| 操作权限 | 该角色能执行的具体动作（增删改查导出等） | `tenant_role_permissions` |

配置**按 (tenant_id, role) 隔离**：A 租户改了 `asset_admin` 的菜单，不会影响 B 租户的 `asset_admin`。
解析层（`/user/menus`、数据范围、接口鉴权）优先读取这些租户表，**为空时回退到全局默认角色权限**。

## 4. 操作指南（前端 UI）

1. 用 **system_admin（或 super_admin）** 账号登录。
2. 通过右上角/顶栏的**租户切换器**选择要配置的租户（切换即携 `X-Tenant-ID` 请求头，权限实时刷新）。
3. 进入 **系统 → 租户角色配置**（菜单 key：`/tenant-role-config`，仅 `super_admin`/`system_admin` 可见）。
4. 在面板顶部的**角色下拉框**选择一个标准角色（如 `asset_admin`）。
5. 在三个 Tab 中配置：
   - **菜单权限**：勾选该角色在当前租户下可见的菜单项，保存后即时生效。
   - **数据范围**：单选 `全院(all)` / `本科室(department)` / `仅本人(self)` / `自定义科室(custom)`；选 custom 时填写科室编码列表。
   - **操作权限**：以标签形式勾选该角色可执行的具体动作（如 `asset.add`、`asset.export`）。
6. 点击各 Tab 的**保存**按钮，接口返回「已更新」即生效；该角色在当前租户下的权限缓存会被自动失效。

> 隔离提示：面板顶部会标注「当前租户：XXX」，所有配置只作用于该租户，不会跨租户串扰。

## 5. 接口文档

所有接口均需请求头：
- `Authorization: Bearer <token>`（登录后 JWT）
- `X-Tenant-ID: <tenantId>`（指定操作所作用的租户；**普通 system_admin 只能操作自己所属的租户**，super_admin 可额外用查询参数 `?tenant_id=` 指定任意租户）

角色白名单（12 个标准角色）：
`super_admin, system_admin, asset_admin, department_admin, metrology_admin, quality_admin, maintenance_admin, maintenance_engineer, acceptance_admin, transfer_admin, inventory_admin, user`
> 注：`super_admin` 角色的配置**仅 super_admin 可改**。

---

### 5.1 列出标准角色

```
GET /api/tenant-role-config/roles
```

响应示例：
```json
{
  "success": true,
  "data": [
    { "role_code": "system_admin", "role_name": "系统管理员",
      "description": "本机构系统负责人…", "is_system_role": 1, "is_active": 1 },
    { "role_code": "asset_admin",   "role_name": "资产管理员",   "is_system_role": 1, "is_active": 1 }
  ]
}
```

---

### 5.2 查询角色菜单可见性（当前租户）

```
GET /api/tenant-role-config/roles/:role/menus
```

响应示例：
```json
{
  "success": true,
  "tenant_id": 2,
  "data": [
    { "menu_key": "/acceptance", "is_visible": 1 },
    { "menu_key": "/ai-assistant", "is_visible": 1 }
  ]
}
```

---

### 5.3 批量设置角色菜单可见性（当前租户）

```
PUT /api/tenant-role-config/roles/:role/menus
Content-Type: application/json
```

请求体：
```json
{
  "menus": [
    { "menu_key": "/acceptance", "is_visible": 1 },
    { "menu_key": "/ai-assistant", "is_visible": 0 }
  ]
}
```
- 整体 upsert，已存在则更新 `is_visible`，并刷新权限缓存。

响应示例：
```json
{ "success": true, "message": "菜单权限已更新", "tenant_id": 2, "count": 2 }
```

---

### 5.4 查询角色数据范围（当前租户）

```
GET /api/tenant-role-config/roles/:role/data-scope
```

响应示例：
```json
{ "success": true, "tenant_id": 2,
  "data": { "data_scope": "all", "custom_department_codes": null } }
```

---

### 5.5 设置角色数据范围（当前租户）

```
PUT /api/tenant-role-config/roles/:role/data-scope
Content-Type: application/json
```

请求体：
```json
{
  "data_scope": "custom",
  "custom_department_codes": ["DEP000100", "DEP1", "DEP2"]
}
```
- `data_scope` 取值：`all` | `department` | `self` | `custom`（非法值返回 400）。
- `custom` 时 `custom_department_codes` 为科室编码数组。

响应示例：
```json
{ "success": true, "message": "数据范围已更新", "tenant_id": 2 }
```

---

### 5.6 查询角色操作权限（当前租户）

```
GET /api/tenant-role-config/roles/:role/permissions
```

响应示例：
```json
{
  "success": true, "tenant_id": 2,
  "data": ["add_assets", "ai.assistant", "asset.add", "asset.export", "asset.view_all", "department.view"]
}
```

---

### 5.7 设置角色操作权限（当前租户，整体替换）

```
PUT /api/tenant-role-config/roles/:role/permissions
Content-Type: application/json
```

请求体：
```json
{ "permissions": ["asset.add", "asset.view_all", "asset.export"] }
```
- 整体替换：先 `DELETE` 该 (tenant_id, role) 下旧记录，再批量 `INSERT`。
- 缓存随角色+租户失效。

响应示例：
```json
{ "success": true, "message": "操作权限已更新", "tenant_id": 2, "count": 3 }
```

---

### 5.8 错误码速查

| HTTP | 场景 | message |
|------|------|---------|
| 400 | 缺 `menus`/`permissions` 数组、无效 `data_scope`、无效角色代码、无租户上下文 | 对应中文提示 |
| 401 | 未带/过期 token | （authenticate 中间件统一返回） |
| 403 | 非 system_admin（或试图改 super_admin 但自身不是 super_admin） | `只有系统管理员可以执行此操作` / `无权配置超级管理员角色` |
| 500 | 数据库异常 | `查询/更新*失败` + `error` 字段 |

## 6. 数据模型

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `tenant_role_menus` | `tenant_id, role, menu_key, is_visible` | 角色在某租户下的菜单可见性（唯一键 tenant_id+role+menu_key） |
| `tenant_role_data_scopes` | `tenant_id, role, data_scope, custom_department_codes` | 角色在某租户下的数据范围 |
| `tenant_role_permissions` | `tenant_id, role, permission_key` | 角色在某租户下的操作权限（唯一键 tenant_id+role+permission_key） |

三张表均以 `(tenant_id, role)` 为隔离维度；迁移脚本已预置 12 角色 × 3 活跃租户的默认值。

---

## 7. 权限解析优先级（运行时）

```
请求进入
  → authenticate 解析 JWT，按 X-Tenant-ID 重新查询该用户在该租户下的角色
  → 菜单: 读 tenant_role_menus，空 → 回退 role_menu_permissions（全局）
  → 数据范围: 读 tenant_role_data_scopes，空 → 回退角色默认
  → 接口鉴权: 读 tenant_role_permissions，空 → 回退角色默认 permission 集合
  → 失败默认关闭（fail-closed）
```

切换租户只需前端带不同的 `X-Tenant-ID`，**无需重新登录**，权限即时按新租户生效（满足需求 ②③）。

## 9. 注意事项

1. **只改当前租户**：所有写操作作用于 `X-Tenant-ID` 指定的租户，不会跨租户串扰。
2. **super_admin 特殊**：仅 super_admin 能改 `super_admin` 角色，且可借 `?tenant_id=` 管理任意租户；普通 system_admin 只能管自己所属租户。
3. **回退机制**：某租户某角色未做任何覆盖配置时，自动使用全局默认角色权限，保证新租户开箱即用。
4. **缓存**：写入后会失效对应 (role, tenant) 的权限缓存，下一次请求即生效，无需重启服务。
5. **既有告警**：`pm-scheduler 调度器启动失败` 与本次功能无关，忽略即可。
