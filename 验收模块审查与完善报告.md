# 验收模块审查与完善报告（2026-07-12）

> 复核范围：验收模块前端 7 个页面 + `acceptanceAPI` + 后端两套验收接口
> 改动文件：前端 8 个、后端控制器 1 个
> 配套：后端已重启（端口 5183），前端为 Vite HMR 自动生效

## 一、模块架构核查（重要背景）

验收模块存在**两套并行后端**，前端分别调用：

| 能力 | 前端 API | 后端文件 | 路由前缀 |
|------|----------|----------|----------|
| 验收**记录** CRUD / 检查清单 / 文件 / 记录统计 | `maintenance.js → acceptanceAPI` | `backend/routes/acceptance.js` | `/api/acceptance/*` |
| 验收**申请**工作流 / 模板 / 报告 / 提醒 / 申请统计 | `AcceptanceApplicationForm` 等直接调 `api` | `backend/modules/acceptance-management/` | `/api/acceptance-management/*` |

- 验收**申请**状态枚举：`草稿/待审批/审批中/已批准/已拒绝/已撤回/已完成`
- 验收**记录**状态枚举：`待验收/验收中/已验收/验收不合格`
- 注意：`acceptance-management` 模块未提供验收记录的创建接口，其统计只读 `asset_acceptance_records`，数据由 legacy 路由写入，属双系统并存的遗留结构（本次未做合并，仅修复功能性 bug）。

## 二、缺陷与修复清单

### P0（阻断性 / 数据污染）

| # | 文件 | 问题 | 修复 |
|---|------|------|------|
| 1 | `api/domains/maintenance.js` | `acceptanceAPI` 缺失 6 个被页面调用的方法（`getChecklist`/`getChecklistStats`/`initChecklist`/`updateChecklistItem`/`passAllChecklist`/`getAssetFillInfo`），导致验收详情页 `Promise.all` 同步抛 `TypeError` → **整页无法加载** | 补齐 6 个方法，路径与 `backend/routes/acceptance.js` 路由逐一对应 |
| 2 | `AcceptanceForm.jsx` | 编辑逻辑依赖调用方**未传入**的 `realRecord` 属性：编辑时表单不回填 + `onFinish` 走 `create` 分支 → **"编辑验收记录"实际变成"新建重复记录"**（数据污染） | 改为以 `record?.id` 判编辑；路由编辑时 `useParams().id` 拉取完整记录回填；列表弹窗用传入的整条 `currentRecord` 回填 |
| 3 | `AcceptanceStatistics.jsx` | 概览字段路径读错：读 `overview.statusDistribution`/`overview.totalRecords` 等**顶层**字段，但后端返回**嵌套**在 `overview.records`/`overview.applications` 下 → 验收记录总数、申请总数、状态分布、部门分布**恒为 0/空** | 改为 `overview.records.total`、`overview.records.statusDistribution`、`overview.applications.total` 等；合格率/模板数路径原本正确保留 |

## 三、验证结果

- **语法**：8 个前端文件 esbuild 转换全部通过；后端控制器 `node --check` 通过。
- **后端重启**：`node server.js` 重启成功（端口 5183，`/api/health` → `database:ok`）。
- **统计接口实测**（tenant 2）：
  - 无日期过滤：`records.total=4`、`records.statusDistribution=[{待验收:4}]`、`applications.total=0`、`passRate=0`、`templateCount=19`（嵌套结构与前端修复后的读取路径**完全吻合**）。
  - 日期过滤 `2020` 区间 → `records.total=0`（过滤生效）；`2026` 区间 → 趋势返回 `2026-02` 共 4 条（区间生效）。

## 四、遗留 / 待办（非阻断）

1. 两套验收后端（`/api/acceptance/*` 与 `/api/acceptance-management/*`）并存，建议后续统一数据源与权限模型（与资产调配模块同样的债务）。
2. `AcceptanceStatistics` 的"验收合格率"口径为 `已验收/(已验收+验收不合格)`，未把"验收中/待验收"计入分母，与趋势表的"合格率"口径不同，属业务定义问题，未改动。
3. `AcceptanceApplicationForm` 的 `applicant_name` 为只读展示，后端创建/更新时仍以登录人覆盖，提交该字段无副作用（已兼容）。
