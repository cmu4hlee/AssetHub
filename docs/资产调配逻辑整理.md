# 资产调配逻辑整理

本文档整理当前项目中**资产调配**相关的数据表、接口、前端页面与业务流程，便于维护与后续统一改造。

---

## 一、概述：两套调配体系

当前资产调配存在**两套并行流程**，共用资产状态「调配中」与「在用」，且互斥检查（同一资产不能同时有待审批的申请与未完成的调配记录）。

| 维度 | 流程一：调配申请 | 流程二：调配记录 |
|------|------------------|------------------|
| **数据表** | `asset_transfer_requests` | `transfer_records` |
| **业务含义** | 用户/科室发起申请 → 管理员审批 → 通过即完成 | 填单创建调配 → 审批 → 再「完成」才更新资产 |
| **状态** | pending / approved / rejected | 待审批 / 已批准 / 已完成 / 已取消 |
| **主要入口** | 资产详情页、菜单「调配申请」、**AI 助手（已统一）** | 仅 POST /transfer（菜单无单独入口） |

## 二、数据表

### 2.1 asset_transfer_requests（调配申请）

- **用途**：记录「提交调配申请」及其审批结果。
- **关键字段**（以当前代码为准）：  
  `id`, `tenant_id`, `asset_code`, `asset_name`, `current_department`, `target_department`, `reason`, `applicant`, `status`(pending/approved/rejected), `approved_by`, `approved_at`, `comment`, `created_at`, `updated_at`
- **创建**：脚本 `backend/scripts/create-asset-transfer-requests-table.js`（建表时曾有 `asset_id`/`asset.code` 等，实际运行中多为 `asset_code` + `tenant_id`）。

### 2.2 transfer_records（调配记录）

- **用途**：记录「调配单」从创建到审批、完成的完整过程。
- **关键字段**（init.sql + 后续迁移）：  
  `id`, `tenant_id`, `transfer_no`, `asset_code`, `from_department`, `to_department`, `from_location`, `to_location`, `from_person`, `to_person`, `transfer_date`, `transfer_reason`, `created_by`, `remark`, `status`(待审批/已批准/已完成/已取消), `approver`, `approve_date`, `created_at`, `updated_at`
- **说明**：init.sql 中为 `asset_id`，当前路由已按 `asset_code` 使用；若表结构已迁移则以库表为准。

## 三、流程一：调配申请（asset_transfer_requests）

### 3.1 提交申请

- **接口**：`POST /assets/:id/transfer-apply`
- **路由**：`backend/routes/assets.js`（`:id` 可为资产 id 或 asset_code）
- **入参**：`target_department`, `reason`
- **逻辑概要**：
  1. 校验目标部门、原因非空；校验资产存在且租户一致。
  2. 权限：非系统管理员只能为自己**管理科室**的资产申请调配。
  3. 资产状态不能为「调配中」；不能存在同资产 pending 的 `asset_transfer_requests`；不能存在同资产「待审批/已批准」的 `transfer_records`。
  4. 目标部门不能与当前部门相同。
  5. 插入 `asset_transfer_requests`（status=pending），资产状态改为「调配中」，写 `asset_change_logs`。
- **前端**：
  - 菜单「调配申请」页：`TransferForm.jsx` → `assetAPI.applyTransfer(asset_code, data)`，即本接口。
  - 资产详情页若有「提交调配申请」按钮，也应调此接口。

### 3.2 获取调配申请列表

- **接口**：`GET /assets/transfer-requests`
- **路由**：`backend/routes/assets.js`
- **查询参数**：`page`, `pageSize`, `status`（空则默认待审批+已审批，不包含已拒绝；`all` 表示全部）
- **权限**：非系统管理员仅能看自己管理科室相关申请（按 current_department / target_department 与 managed_departments 过滤）。
- **前端**：「调配申请处理」页 `TransferRequestList.jsx` → `assetAPI.getTransferRequests(params)`。

### 3.3 审批申请（通过/拒绝）

- **接口**：`POST /assets/transfer-requests/:id/approve`
- **路由**：`backend/routes/assets.js`，内部调用 `backend/services/transfer-approval-service.js` 的 `approveTransferRequest(connection, opts)`。
- **权限**：仅 super_admin / system_admin。
- **入参**：`approved`（boolean）, `comment`（可选）
- **逻辑概要**（见 `transfer-approval-service.js`）：
  - **通过**：校验资产存在且状态为「调配中」→ 更新资产 `department`、`department_new`、`status='在用'` → 写 `asset_change_logs`（部门调配）→ 更新申请状态为 approved。
  - **拒绝**：更新申请状态为 rejected；若该资产为「调配中」且无其他 pending 申请且无其他未完成 transfer_records，则资产状态恢复为「在用」。
- **前端**：`TransferRequestList.jsx` → `assetAPI.approveTransferRequest(id, { approved, comment })`；AI 助手内「通过/拒绝调配」也调用同一审批逻辑（见下）。

## 四、流程二：调配记录（transfer_records）

### 4.1 列表与详情（实际读的是申请表）

- **列表**：`GET /transfer` → `backend/routes/transfer.js`  
  **实现**：从 `asset_transfer_requests` 表查询，并映射为前端所需字段（如 `from_department`=current_department, `to_department`=target_department, `transfer_no`=SQ+id 等）。  
  **说明**：前端「调配记录」页展示的其实是「调配申请」数据，并非 `transfer_records` 表。
- **详情**：`GET /transfer/:id` → 同上，按 id 查 `asset_transfer_requests`。

### 4.2 创建调配记录

- **接口**：`POST /transfer`
- **路由**：`backend/routes/transfer.js`
- **入参**：`transfer_no`, `asset_code`, `from_department`, `to_department`, `transfer_date` 必填；`from_location`, `to_location`, `from_person`, `to_person`, `transfer_reason`, `remark` 可选。
- **逻辑概要**：
  1. 校验调配单号在同租户内唯一（查 `transfer_records`）；资产存在且非「调配中」；无同资产 pending 的 `asset_transfer_requests`；无同资产「待审批/已批准」的 `transfer_records`。
  2. **系统管理员**：直接更新资产部门/位置/责任人/状态为「在用」，写 change_log，并插入 `transfer_records` 一条 status=「已完成」。
  3. **非系统管理员**：资产状态改为「调配中」，插入 `transfer_records` 一条 status=「待审批」。
- **前端**：AI 助手在「调配」意图下填单后调用 `transferAPI.createTransfer(transferPayload)`；菜单「调配申请」页（TransferForm）走的是申请流程，不调用本接口。

### 4.3 审批调配（transfer_records）

- **接口**：`PUT /transfer/:id/approve`
- **路由**：`backend/routes/transfer.js`
- **入参**：`status`（已批准/已取消）, `approve_date`（可选）
- **逻辑**：仅处理 `transfer_records` 中 id 对应记录；状态须为「待审批」；批准则更新为「已批准」，拒绝/取消则更新为「已取消」并视情况恢复资产「在用」（需无其他未完成申请/记录）。

### 4.4 完成调配（transfer_records）

- **接口**：`PUT /transfer/:id/complete`
- **路由**：`backend/routes/transfer.js`
- **逻辑**：仅处理 `transfer_records`；状态须为「已批准」；资产须为「调配中」；无同资产其他未完成调配记录；然后更新记录为「已完成」，更新资产部门/位置/责任人/状态为「在用」，写 change_log。

### 4.5 删除「调配记录」（实际删的是申请）

- **接口**：`DELETE /transfer/:id`
- **路由**：`backend/routes/transfer.js`  
  **实现**：按 id 删除的是 `asset_transfer_requests` 表记录；若申请为 pending/rejected，会视情况将资产状态恢复为「在用」。
- **前端**：「调配记录」页 `TransferList.jsx` 的删除调用本接口，因此实际删除的是「调配申请」数据，与列表数据一致（列表本就来自 asset_transfer_requests）。

## 五、前端页面与 API 对应关系

| 页面 | 路由 | 列表/数据来源 | 提交/操作 API |
|------|------|----------------|----------------|
| 调配记录 | `/transfer` | `GET /transfer` → asset_transfer_requests | 删除：`DELETE /transfer/:id`（删申请） |
| 调配申请（填单） | `/transfer/new` | - | `POST /assets/:id/transfer-apply`（asset_code 作为 id） |
| 调配申请处理 | `/transfer/requests` | `GET /assets/transfer-requests` | `POST /assets/transfer-requests/:id/approve` |

- **POST /transfer**（创建 transfer_records）：目前仅 AI 助手内「调配」意图填单后调用；菜单里没有单独「创建调配记录」的入口（与「调配申请」表单不同）。

## 六、互斥与一致性

- **同一资产**在任意时刻：
  - 不能存在多条「未结束」的调配申请或调配记录；
  - 提交申请前会检查：资产非「调配中」、无 pending 的 asset_transfer_requests、无「待审批/已批准」的 transfer_records；
  - 创建 transfer_records 前同样会检查：无 pending 的 asset_transfer_requests、无未完成的 transfer_records。
- **资产状态**：「调配中」仅在申请/记录未完成时保持；审批通过（申请）或完成调配（记录）后改为「在用」；拒绝或取消时若无其他未完成项则恢复「在用」。
- **transfer-approval-service** 拒绝时恢复资产前，会同时检查 `asset_transfer_requests` 与 `transfer_records` 是否还有未完成项；transfer 路由在审批拒绝、完成、删除时也有类似检查。

## 七、AI 助手中的调配（已统一为申请制）

- **意图**：`transfer`（资产调配）、`transfer_approve`（审批调配）。
- **待办**：打开对话或用户说「待办/待审批」时，会拉取待审批报修 + 待审批调配单；调配单来自 `asset_transfer_requests`（status=pending）。
- **调配填单**：用户说「调配/资产调配」后，收集**资产编号、调入部门、调配原因**（必填）；选填调出部门（展示用）。提交时调用 **`assetAPI.applyTransfer(asset_code, { target_department, reason })`**，即 **POST /assets/:id/transfer-apply**（走 asset_transfer_requests 申请制），与菜单「调配申请」、审批流程一致。
- **审批调配**：用户说「通过第一条/通过调配 5」等，调用 `approveTransferRequest`，即审批 **asset_transfer_requests**（与「调配申请处理」页一致），仅系统管理员可执行。

## 八、问题与建议

1. **两套流程并存**  
   - 申请流程：申请 → 审批即完成，表 `asset_transfer_requests`。  
   - 记录流程：创建调配单 → 审批 → 再「完成」，表 `transfer_records`。  
   菜单「调配记录」列表实际展示的是申请表数据。**AI 助手已统一为申请制**（提交走 POST /assets/:id/transfer-apply），与待办、审批一致。  
   **建议**：若不再使用「调配记录」流程，可考虑仅保留申请制，或将 /transfer 列表与删除明确只针对 asset_transfer_requests。

2. **GET /transfer 与 DELETE /transfer/:id 语义**  
   - GET /transfer 读的是 asset_transfer_requests，返回字段名做了映射（如 from_department）。  
   - DELETE /transfer/:id 删的也是 asset_transfer_requests。  
   但 POST /transfer、PUT /transfer/:id/approve、PUT /transfer/:id/complete 操作的是 transfer_records，同一路由前缀下混用两张表，维护成本高。  
   **建议**：要么将「调配记录」页明确改为只展示 transfer_records（需新增按 transfer_records 的列表接口），要么将 /transfer 前缀统一为只处理 asset_transfer_requests，另一套用 /transfer-records 等前缀。

3. **租户与权限**  
   - 提交申请、获取申请列表、审批申请均已做租户过滤；transfer 路由各接口也有 addTenantFilter。  
   - 建议在文档或代码注释中明确：所有调配相关接口均需登录且带租户上下文（selectedEnterprise 或 user.tenant_id）。

4. **表结构**  
   - asset_transfer_requests 建表脚本中存在字段名笔误（如 asset.code）；若表已通过迁移改为 asset_code、tenant_id，建议在脚本或迁移说明中统一，避免新环境建表失败。

## 十、AI 助手适配说明（改造后）

- **改造内容**：AI 助手内「调配」意图由原先的「调配记录」流程（POST /transfer → transfer_records）改为**调配申请**流程（POST /assets/:id/transfer-apply → asset_transfer_requests）。
- **必填项**：由「资产编号、调出部门、调入部门、调配日期」改为「资产编号、调入部门、调配原因」。调出部门由系统按资产当前部门自动带出，无需用户填写。
- **效果**：用户在 AI 中提交的调配会进入「调配申请处理」列表，与待办、审批调配一致，避免两套数据源混用。
- **涉及文件**：`backend/services/maintenance-ai-service.js`（必填项、提示词、reason 提取）、`frontend/src/pages/AIMaintenanceManager.jsx`（提交改为 assetAPI.applyTransfer，必填项与文案）。

以上为当前资产调配逻辑的整理，后续若统一为一套流程或调整表与接口，可据此文档做影响范围排查与修改。
