## 二、模块配置与缺失前端组件

（与《项目缺失文件分析报告》一致，此处摘要。）

### 2.1 已修复：asset-ai-assistant

- 模块配置中已将对已删除页面 `AssetQueryAssistant` 的引用改为 `AssetAIAssistant`，并移除了不存在的组件条目。

### 2.2 quality-common 模块

- **现象**：`backend/modules/quality-common/config/module.config.js` 声明了三个前端组件及对应菜单：
  - `PermissionManagement` → `pages/PermissionManagement`
  - `DataDictionary` → `pages/DataDictionary`
  - `LogViewer` → `pages/LogViewer`
- **问题**：`frontend/src/pages/` 下不存在上述三个页面。
- **影响**：若按模块配置生成菜单（如 `/quality/common/permission`、`/quality/common/dictionary`、`/quality/common/log`）并做按需加载，会 404 或加载失败。
- **建议**：新增占位页并跳转到现有「角色与权限」/审计日志，或在模块配置中移除/改写上述菜单与组件引用。

### 2.3 adverse-event 模块

- **现象**：`backend/modules/adverse-event/config/module.config.js` 的 `components` 中包含：
  - `AdverseEventWorkflowList`、`AdverseEventAnalysisDashboard`、`PreventiveMeasuresList`、`RootCauseAnalysisForm`
- **问题**：对应页面/组件在 `frontend/src/pages/` 与 `frontend/src/components/` 下均不存在。
- **影响**：当前不良事件主流程（List/Form/Detail）正常；若菜单或后续功能按配置加载上述组件会报错。
- **建议**：短期可从配置中移除上述组件引用；若计划做工作流/分析/预防措施/根因分析，再补齐页面与组件。

## 三、占位或未实现功能

### 3.1 消息平台集成（微信/飞书/钉钉/短信）

- **位置**：`frontend/src/pages/IntegrationChannelPage.jsx`
- **现象**：页面为统一占位，文案写明「可在后续接入具体绑定、通知模板与回调配置能力」；四个路由（`/integration/wechat-binding`、`sms-notification`、`feishu-binding`、`dingtalk-binding`）均渲染该页。
- **后端**：模块配置中声明了 `GET/POST /api/integration/channels`、`/api/integration/channels/:channel`，但 `backend/server.js` 未挂载任何 `/api/integration` 路由。
- **结论**：功能为占位，前后端均未实现具体集成逻辑。

### 3.2 工作流编辑与执行

- **前端**：`frontend/src/components/WorkflowEditor.jsx` 存在，但**未被任何页面 import 或使用**，属于死组件。
- **API**：组件内写死 `API_BASE = 'http://localhost:3010/api'`，请求 `GET/POST/PUT /api/workflows`、`POST /api/workflows/:id/execute`。
- **后端**：主项目已删除 `backend/routes/asset-workflows.js`，未挂载 `/api/workflows`；仅保留 `backend/services/asset-workflow.service.js`（无 HTTP 层）。
- **代理**：`frontend/vite.config.js` 将 `/api/workflows`、`/api/workflow-executions`、`/api/nodes` 代理到 `http://localhost:3010`，依赖外部工作流服务。
- **结论**：工作流功能不完整：前端组件未接入、主后端无路由、依赖 3010 外部服务且未在文档中说明。

### 3.3 聊天室（chatAPI）

- **前端**：`frontend/src/api/domains/platform.js` 中定义了 `chatAPI`（getRooms、createRoom、getRoomDetails、join/leave、sendMessage、getRoomMessages、getOnlineUsers、userOffline），并在 `utils/api.js` 中导出。
- **使用情况**：全局搜索未发现任何页面或组件调用 `chatAPI`。
- **后端**：主 backend 无 `/api/chat-rooms` 或类似路由。
- **结论**：聊天室为预留 API 定义，无后端、无前端使用，属死代码/预留扩展。

### 高优先级（影响配置一致性或易触发 404）

### 中优先级（占位与体验）

3. **消息平台集成**：若近期不做微信/飞书/钉钉/短信集成，可在集成页注明“规划中”或隐藏菜单；若做，则需实现 `/api/integration` 及前端表单/配置。

### 低优先级（清理与文档）

5. **chatAPI**：若无聊天室规划，可从 `platform.js` 与 `utils/api.js` 中移除 chatAPI，减少误导。
6. ~~**quality-control-service**~~：已更新 getExpiringQualityControlRecords 注释。

完成上述项后，建议再跑一遍前端构建与后端启动，并做一次主要菜单与关键 API 的冒烟测试，确认无 404 与未实现调用。

## 八、继续检查（后续修复）

### 8.1 前端「资产管理助手」弹窗 URL（已修复）

- **位置**：`frontend/src/components/Layout.jsx`、`frontend/src/pages/AIAssistantRedirect.jsx`
- **现象**：点击菜单「资产管理助手」或从重定向打开时，弹窗使用硬编码 `http://localhost:8000/#/zcgl`，与当前前端端口（如 13579）及 BrowserRouter 路径不一致，易导致弹窗打开错误页面。
- **处理**：已改为使用 `window.location.origin` + 当前 pathname 拼接 `/zcgl`，弹窗始终打开当前站点下的 `/zcgl`。

### 8.2 后端平台 AI 接口说明（已下线）

- **位置**：已移除历史平台代理路由与挂载。
- **说明**：当前 AI 助手页面采用 `/chat-api/api/chat` 直连方式，不再经过后端平台代理层。
