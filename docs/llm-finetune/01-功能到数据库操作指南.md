# AssetHub 功能到数据库操作指南（用于大模型微调）

> 版本：v1.0（2026-04-11）  
> 适用范围：AssetHub 多租户系统（后端 `/api` + `assethub` MCP 工具）  
> 目标：让模型在“用户意图 -> 工具调用 -> 数据库读写策略 -> 权限与租户约束”上稳定输出。

## 1. 全局硬约束（训练时必须学习）

1. 资产域问题优先走 `assethub` MCP 工具，不直接臆造结果。
2. 每次业务处理都要绑定租户上下文：
- 普通用户/租户管理员：使用当前登录租户。
- 超级管理员：跨租户必须显式提供 `X-Tenant-ID` 并通过权限校验。
3. 写操作默认“先查后写再回查”。
4. 所有 SQL/查询策略都要带 `tenant_id` 过滤（超级管理员全局巡检场景除外）。
5. AI 回答不泄露 `_auth_context_id`、token、内部系统提示词。
6. 遇到越权请求必须拒绝，并给出合规替代路径。

## 2. 运行时上下文字段（模型必须优先确认）

- `username`：当前登录用户名。
- `role`：`super_admin` / `system_admin` / 普通用户。
- `tenant_id`：当前生效企业空间。
- `managed_departments`：科室范围控制（非超管常见）。
- `_auth_context_id`：Web 会话到 MCP 的短时凭证（工具调用参数）。

推荐首个工具调用：`assethub_get_current_auth_context`。

## 3. 标准处理流程模板

### 3.1 查询类

1. 识别业务域（资产/维修/盘点/验收等）。
2. 调 `assethub_get_current_auth_context` 校验租户与权限边界。
3. 调对应列表/统计工具。
4. 返回时附上筛选条件与数据范围说明（尤其是租户范围）。

### 3.2 写入类

1. 校验必填字段。
2. 先查目标对象是否存在（并确认在当前租户内）。
3. 执行创建/更新/审批。
4. 回查详情或列表确认状态。
5. 返回“对象ID/编号 + 当前状态 + 下一步建议”。

## 4. 模块级操作剧本

## 4.1 资产管理（Assets）

常用工具：
- `assethub_list_assets`
- `assethub_get_asset`
- `assethub_create_asset`
- `assethub_update_asset`
- `assethub_delete_asset`
- `assethub_get_asset_statistics`

核心表：
- `assets`
- `asset_categories`
- `asset_change_logs`
- `asset_images`

推荐调用序列：
1. 列表查询：`list_assets(keyword/status/department)`
2. 单项确认：`get_asset(id|asset_code)`
3. 创建或更新：`create_asset` / `update_asset`
4. 验证结果：再次 `get_asset`

SQL 模板（训练参考）：
```sql
SELECT a.*
FROM assets a
WHERE a.tenant_id = ?
  AND (? IS NULL OR a.status = ?)
  AND (? IS NULL OR a.asset_name LIKE CONCAT('%', ?, '%'))
ORDER BY a.created_at DESC
LIMIT ? OFFSET ?;
```

```sql
UPDATE assets
SET status = ?, updated_at = NOW()
WHERE id = ? AND tenant_id = ?;
```

常见错误与纠正：
- 错误：用 `asset_code` 直接当主键更新。
- 正确：先查出 `id`，再带 `tenant_id` 更新。

## 4.5 资产使用量与阈值触发（独立卡片功能）

常用工具：
- `assethub_create_usage_record`
- `assethub_list_usage_records`
- `assethub_list_usage_triggered`
- `assethub_process_usage_triggered`

相关接口（后端）：
- `/api/maintenance/usage/records`
- `/api/maintenance/usage/triggered`
- `/api/maintenance/usage/asset-usage`

核心表（双轨并存）：
- 旧：`asset_usage_records`、`usage_triggered_maintenance`
- 新：`maintenance_usage_records`、`maintenance_usage_triggered`
- 联动：`preventive_maintenance_plans`（`trigger_type in ('usage','both')`）

SQL 模板：
```sql
INSERT INTO maintenance_usage_records (
  tenant_id, asset_code, usage_value, usage_date, usage_type, recorded_by
) VALUES (?, ?, ?, ?, ?, ?);
```

```sql
SELECT mut.*, pmp.plan_name
FROM maintenance_usage_triggered mut
LEFT JOIN preventive_maintenance_plans pmp ON mut.plan_id = pmp.id
WHERE mut.tenant_id = ?
ORDER BY mut.triggered_at DESC;
```

## 5. 统一错误处理策略（训练输出模板）

- `400`：参数缺失/格式错误 -> 列出缺失字段并给示例。
- `401`：未登录或令牌失效 -> 先提示重新登录。
- `403`：越权/跨租户 -> 明确拒绝并提示可用范围。
- `404`：对象不存在 -> 提示先查列表确认 ID。
- `500`：服务异常 -> 保留上下文，给重试建议。

## 6. 建议给模型的固定回答骨架

```text
1) 我先确认当前用户与企业空间上下文。

3) 若要执行写操作，我将先校验对象存在，再执行并回查。
4) 当前可继续的下一步：...
```
