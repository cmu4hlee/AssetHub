分析时间：2026-02-25  
分析范围：前后端入口与路由、模块配置引用、已删除文件残留引用。

## 二、必须修复：资产 AI 助手模块

**现象**：`backend/modules/asset-ai-assistant/config/module.config.js` 中：

- `menu_routes[0].component` 为 `'AssetQueryAssistant'`
- `frontend_config.components` 包含 `{ name: 'AssetQueryAssistant', path: 'pages/AssetQueryAssistant', export: 'default' }`

**问题**：前端页面 `frontend/src/pages/AssetQueryAssistant.jsx` 与 `AssetQueryAssistant.css` 已删除（git 状态为 AD）。任何按模块配置做组件校验或按路径动态加载的逻辑都会指向不存在的文件。

**影响**：当前 App.jsx 中 `/zcgl` 路由写死为 `<AssetAIAssistant />`，因此访问 `/zcgl` 仍可正常打开；但模块配置与前端实现不一致，且若将来改为按模块配置动态渲染该菜单对应组件，会加载失败。

**建议**：

- 将 `menu_routes` 中 path `/zcgl` 的 `component` 改为 `'AssetAIAssistant'`。
- 从 `frontend_config.components` 中删除 `AssetQueryAssistant` 条目，仅保留 `AssetAIAssistant`。

---

## 三、建议修复：quality-common 模块

**现象**：`backend/modules/quality-common/config/module.config.js` 声明了三个前端组件：

- `PermissionManagement` → `pages/PermissionManagement`
- `DataDictionary` → `pages/DataDictionary`
- `LogViewer` → `pages/LogViewer`

**问题**：在 `frontend/src/pages/` 下不存在上述三个页面文件。

**影响**：若系统按该模块配置生成菜单（如 `/quality/common/permission`、`/quality/common/dictionary`、`/quality/common/log`）并做按需加载，会 404 或加载报错。

**建议**（任选其一或组合）：

1. **占位页**：新增简单的 `PermissionManagement.jsx`、`DataDictionary.jsx`、`LogViewer.jsx`，内部可跳转到现有「角色与权限」或审计日志等页面；
2. **改配置**：若当前产品不需要这些菜单位置，可在模块配置中移除或注释上述 menu_routes/components，避免引用不存在的路径；
3. **复用现有页**：若「权限管理」即指现有「角色与权限」，可把配置中的 `PermissionManagement` 改为指向 `RolePermissionManagement` 的路径（并确保导出名与配置一致）。

---

## 四、建议修复：adverse-event 模块

**现象**：`backend/modules/adverse-event/config/module.config.js` 的 `frontend_config.components` 中包含：

- `AdverseEventWorkflowList` → `pages/AdverseEventWorkflowList`
- `AdverseEventAnalysisDashboard` → `pages/AdverseEventAnalysisDashboard`
- `PreventiveMeasuresList` → `pages/PreventiveMeasuresList`
- `RootCauseAnalysisForm` → `components/RootCauseAnalysisForm`

**问题**：在 `frontend/src/pages/` 与 `frontend/src/components/` 下均未找到以上文件。

**影响**：当前不良事件菜单只配置了 AdverseReactionList/Form/Detail，主流程不受影响；但若菜单或后续功能扩展使用上述组件（如工作流列表、分析看板、预防措施列表、根因分析表单），会因文件缺失而报错。

**建议**：

- 若近期不会做这些子功能：在模块配置中从 `components` 中移除上述四项，或改为注释，避免被动态加载逻辑引用；
- 若计划实现：按配置路径补齐对应页面/组件。

---

## 五、可清理：seed_modules 中的 asset-workflows

**现象**：`backend/database/migrations/seed_modules.js` 中，asset-management 的 `frontend_config.menu_prefixes` 仍包含 `'/asset-workflows'`。

**问题**：后端路由 `backend/routes/asset-workflows.js` 已删除，且 `server.js` 未挂载该路由，属于历史配置残留。

**建议**：在 `seed_modules.js` 的 asset-management 的 `frontend_config.menu_prefixes` 中删除 `'/asset-workflows'`，仅保留 `/assets` 等实际使用的前缀。

- **App.jsx 中直接 import 的页面**：均在 `frontend/src/pages/` 或子目录下存在；lazy 的 TechnicalDocuments*、AssetShareUpload 等对应文件也存在。
- **已删除且未被引用的**：如 `tenantModuleConfigApi.js`、`EnhancedPermissionManagement.jsx`、`DashboardConfigManager.jsx` 等，当前代码中未发现残留引用，无需恢复文件。
