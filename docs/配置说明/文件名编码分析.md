## 数据流程分析

### 1. 前端阶段（JavaScript）

- **位置**：`MaintenanceLogForm.jsx` 第 233 行
- **代码**：`formData.append('file', file, file.name)`
- **file.name 编码**：JavaScript 字符串，UTF-16（内部）/ UTF-8（传输）
- **状态**：✅ 正常，JavaScript 字符串是 UTF-8 编码

### 2. 浏览器编码阶段

- **HTTP 协议**：multipart/form-data
- **文件名编码规范**：
  - RFC 2047：`=?charset?encoding?encoded-text?=`
  - RFC 2231：`filename*=charset'lang'encoded-text`
  - RFC 5987：`filename*=UTF-8''encoded-text`（推荐）
- **浏览器行为**：
  - 现代浏览器（Chrome、Firefox、Safari）通常使用 RFC 5987 格式
  - 格式：`Content-Disposition: form-data; name="file"; filename="原始文件名"; filename*=UTF-8''URL编码的文件名`
- **可能的问题**：如果浏览器使用旧格式或编码不正确

### 3. 网络传输阶段

- **HTTP 请求**：通过 axios 发送
- **Content-Type**：`multipart/form-data; boundary=...`
- **状态**：✅ 正常，HTTP 协议支持二进制传输

### 4. multer 解析阶段（关键问题点）

- **库版本**：multer 2.0.2
- **解析方式**：使用 `busboy` 库解析 multipart/form-data
- **问题根源**：
  - multer/busboy 在解析文件名时，可能没有正确处理 RFC 5987 格式
  - 如果浏览器发送了 `filename` 和 `filename*`，multer 可能优先使用 `filename`（可能是 ISO-8859-1 编码）
  - 如果 `filename*` 存在但格式不正确，multer 可能无法正确解析

### 5. 后端接收阶段

- **位置**：`maintenance.js` 第 1450 行
- **获取方式**：`file.originalname`
- **问题**：此时文件名可能已经是乱码（UTF-8 字节被解释为 Latin1）

## 根本原因

### 核心问题

**multer/busboy 在解析 multipart/form-data 时，如果同时存在 `filename` 和 `filename*`，可能优先使用 `filename`（ISO-8859-1 编码），而不是 `filename*`（UTF-8 编码）。**

### 编码转换链

1. 原始文件名（UTF-8）：`西门子ForceDrive双源CT的VB10版本系统更换数据硬盘后无需重装系统.pdf`
2. 浏览器编码为 RFC 5987：`filename*=UTF-8''%E8%A5%BF%E9%97%A8%E5%AD%90...`
3. multer 错误地使用 `filename`（如果存在）：可能是 ISO-8859-1 编码的版本
4. Node.js 将 ISO-8859-1 字节解释为字符串：`è¥¿é©é©å­©...`
5. 此时字符串已经是乱码，后续的 UTF-8 解码无法恢复

## 解决方案

### 方案1：在 multer 解析前手动解析 Content-Disposition（推荐）

在 multer 处理之前，手动解析 HTTP 请求头，提取正确的 UTF-8 编码文件名。

### 方案2：使用 multer 的配置选项

检查 multer 是否有配置选项可以指定编码处理方式。

### 方案3：使用原始请求体解析

绕过 multer，直接解析 multipart/form-data 请求体。

### 方案4：前端预处理

在前端将文件名进行 Base64 编码，后端解码。

## 推荐解决方案

**方案1 + 方案4 组合**：

1. 前端：除了正常的文件上传，额外发送一个 Base64 编码的文件名字段
2. 后端：优先使用 Base64 解码的文件名，如果不存在，再尝试从 Content-Disposition 解析
