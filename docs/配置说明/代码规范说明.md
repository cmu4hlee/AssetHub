# 代码规范说明

本项目使用 ESLint 和 Prettier 来统一代码风格，确保代码质量和一致性。

## 工具配置

### ESLint
- **前端**: 使用 ESLint 9.x (flat config)
- **后端**: 使用 ESLint 8.x (传统配置)

### Prettier
- 统一代码格式化工具
- 配置在 `.prettierrc` 文件中

## 安装依赖

### 前端
```bash
cd frontend
npm install
```

### 后端
```bash
cd backend
npm install
```

## 使用命令

### ESLint 检查

**前端:**
```bash
npm run lint          # 检查代码问题
npm run lint:fix      # 自动修复可修复的问题
```

**后端:**
```bash
npm run lint          # 检查代码问题
npm run lint:fix      # 自动修复可修复的问题
```

### Prettier 格式化

**前端:**
```bash
npm run format        # 格式化所有代码
npm run format:check  # 检查代码格式（不修改）
```

**后端:**
```bash
npm run format        # 格式化所有代码
npm run format:check  # 检查代码格式（不修改）
```

## 代码规范规则

### 通用规则

1. **缩进**: 使用 2 个空格
2. **引号**: 使用单引号（字符串）
3. **分号**: 语句末尾使用分号
4. **行尾**: 使用 LF (Unix 风格)
5. **行宽**: 最大 100 字符
6. **尾随逗号**: ES5 兼容模式（对象和数组多行时使用）

### JavaScript/JSX 规则

1. **未使用变量**: 警告（允许以下划线开头的变量）
2. **console**: 允许 `console.warn` 和 `console.error`，其他警告
3. **React Hooks**: 遵循 React Hooks 规则
4. **React Refresh**: 仅导出组件时警告

### 后端 Node.js 规则

1. **环境**: Node.js 和 ES2021
2. **未使用变量**: 警告（允许以下划线开头的变量）
3. **console**: 允许 `console.warn`、`console.error` 和 `console.info`
4. **prefer-const**: 优先使用 const
5. **no-var**: 禁止使用 var

## IDE 集成

### VS Code

安装以下扩展：
- ESLint
- Prettier - Code formatter

在 `.vscode/settings.json` 中添加：

```json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  },
  "[javascript]": {
    "editor.defaultFormatter": "esbenp.prettier-vscode"
  },
  "[javascriptreact]": {
    "editor.defaultFormatter": "esbenp.prettier-vscode"
  }
}
```

### WebStorm/IntelliJ IDEA

1. 启用 ESLint: `Settings` → `Languages & Frameworks` → `JavaScript` → `Code Quality Tools` → `ESLint`
2. 启用 Prettier: `Settings` → `Languages & Frameworks` → `JavaScript` → `Prettier`
3. 启用 "On save" 格式化

## 忽略文件

以下文件/目录会被忽略：

- `node_modules/`
- `dist/` / `build/`
- `*.config.js`
- `.env` 文件
- 日志文件

详细配置见 `.eslintignore` 和 `.prettierignore`

## 提交前检查

建议在提交代码前运行：

```bash
# 前端
cd frontend
npm run lint:fix
npm run format

# 后端
cd backend
npm run lint:fix
npm run format
```

## 持续集成

可以在 CI/CD 流程中添加：

```yaml
# 示例 GitHub Actions
- name: Lint
  run: |
    cd frontend && npm run lint
    cd ../backend && npm run lint

- name: Format Check
  run: |
    cd frontend && npm run format:check
    cd ../backend && npm run format:check
```

## 常见问题

### ESLint 报错但代码看起来没问题

1. 检查是否有语法错误
2. 运行 `npm run lint:fix` 自动修复
3. 查看 ESLint 输出中的具体规则

### Prettier 格式化后代码变化很大

这是正常的，Prettier 会统一代码风格。建议：
1. 提交前运行 `npm run format`
2. 配置 IDE 自动格式化
3. 团队统一使用相同的配置

### 某些文件不想被格式化

在 `.prettierignore` 或 `.eslintignore` 中添加文件路径。

## 更新配置

如果需要修改代码规范：

1. **ESLint**: 修改 `.eslintrc.js` (后端) 或 `eslint.config.js` (前端)
2. **Prettier**: 修改 `.prettierrc`

修改后运行 `npm run lint:fix` 和 `npm run format` 更新现有代码。
