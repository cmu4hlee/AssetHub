# 多租户用户角色权限管理系统设计

> 适用项目：AssetHub（医学装备运维管理平台）
> 编写日期：2026-07-12
> 状态：设计方案（待评审 → 实施）

## 0. 设计目标（四条硬需求）

1. **一用户多租户**：一个用户可关联多个租户（企业/医院），在不同租户下拥有不同的角色与权限。
2. **选租户即动态授权**：登录后选择/进入某个租户时，系统按"该用户在该租户下的角色"动态分配权限。
3. **可切换、实时生效**：支持在多个租户间切换，每次切换后权限**实时更新**（无残留旧上下文）。
4. **租户自配、相互独立**：角色与权限可由**租户管理员**配置，不同租户的角色/权限体系**相互独立**。

## 1. 设计原则

- **租户隔离**：所有业务数据、角色定义、权限配置按 `tenant_id` 物理隔离；租户管理员只能管理本租户。
- **系统模板 + 租户实例**：系统预置 12 个标准角色作为**只读模板**；每个租户可克隆并自定义，互不影响。
- **三层权限叠加**：角色基线 → 租户级角色覆盖 → 用户级覆盖（细粒度 allow/deny）。
- **fail-closed**：新增菜单/权限默认对所有角色隐藏，显式授予才可见。
- **token 即上下文**：JWT 内携带当前 `tenant_id`+`role`，切换即重签，杜绝跨租户串数据。
- **向后兼容**：迁移期保留全局 `role_menu_permissions` 作为系统模板与回退源，租户表为空时自动回退。

## 4. 数据模型

### 4.1 复用现有表（不改结构）
- `tenants`、`users`、`menu_definitions`、`tenant_module_configs`、`tenant_module_menus`

### 4.2 保留为系统模板/回退
- `roles`（12 系统角色，`is_system_role=1`，只读模板）
- `role_menu_permissions`（系统模板：role→menu_key，回退源）

### 4.3 新增表（per-tenant 角色体系）

```sql
-- 租户级角色定义（每个租户独立）
CREATE TABLE tenant_roles (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id     INT NOT NULL,
  role_code     VARCHAR(64) NOT NULL,        -- 租户内唯一
  role_name     VARCHAR(128) NOT NULL,
  description   VARCHAR(255),
  is_system_template TINYINT DEFAULT 0,       -- 是否从系统模板克隆而来
  is_active     TINYINT DEFAULT 1,
  created_by    BIGINT,                       -- 创建者 user_id
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_tenant_role (tenant_id, role_code),
  KEY idx_tenant (tenant_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 租户级 角色→菜单 可见性
CREATE TABLE tenant_role_menus (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_role_id BIGINT NOT NULL,
  menu_key      VARCHAR(128) NOT NULL,
  is_visible    TINYINT DEFAULT 0,
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_role_menu (tenant_role_id, menu_key),
  KEY idx_role (tenant_role_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 租户级 角色→数据范围
CREATE TABLE tenant_role_data_scopes (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_role_id BIGINT NOT NULL,
  data_scope    ENUM('all','department','self','custom') NOT NULL DEFAULT 'department',
  custom_department_codes JSON,               -- data_scope=custom 时指定
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_role_scope (tenant_role_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 租户级 角色→操作权限
CREATE TABLE tenant_role_permissions (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_role_id BIGINT NOT NULL,
  permission_key VARCHAR(128) NOT NULL,       -- 如 asset:create, maintenance:approve
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_role_perm (tenant_role_id, permission_key),
  KEY idx_role (tenant_role_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 操作权限目录（全局目录，租户从中选用）
CREATE TABLE tenant_permissions (
  permission_key VARCHAR(128) PRIMARY KEY,
  name           VARCHAR(128) NOT NULL,
  module         VARCHAR(64),
  description    VARCHAR(255),
  sort_order     INT DEFAULT 0
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.4 user_tenant_roles.role 语义（迁移关键）
- 现状：`role` 存系统 `role_code`（如 `maintenance_engineer`）。
- 新方案：新增列 `tenant_role_id BIGINT NULL`。
  - 优先用 `tenant_role_id`（租户自定义角色）；
  - 为空则回退用 `role`（系统模板角色），保证平滑迁移。
- 迁移期两者并存，应用层解析时：`effective_role = tenant_role_id ?? lookupSystemRole(role)`。

## 5. 核心流程

### 5.1 登录与租户选择
1. `POST /api/users/login`（凭 username+password）。
2. 后端查 `user_tenant_roles` 得到该用户所有可访问租户及对应角色 → `enterprises[]`。
3. 若仅 1 个租户：直接签发含该租户+角色的 JWT，返回 token + user。
4. 若多个租户：返回 `enterprises` 列表 + 一个**临时登录 token**（仅含 user_id，无 tenant 上下文，有效期 5 分钟），前端进入 `EnterpriseSelect` 选择。
5. 选定后调 `POST /api/users/switch-tenant { tenant_id }`（见 5.2）换取正式 token。

### 5.2 租户切换（实时权限更新）—— 满足需求②③
```
POST /api/users/switch-tenant
Body: { tenant_id: 2 }
Auth: Bearer <当前 token>
```
后端逻辑：
1. `authenticate` 校验当前 token。
2. 查 `user_tenant_roles WHERE user_id=? AND tenant_id=? AND status='active'`，确认用户在该租户有有效角色；否则 403。
3. 读取该租户下的角色（`tenant_role_id` 优先，否则系统 `role`）。
4. **重签 JWT**：`{ user_id, tenant_id, role, role_type, session_id, exp }`，刷新 `session_id`（使旧上下文缓存失效）。
5. 返回 `{ token, user:{...,tenant_id,role,tenant_name}, enterprises:[...] }`。

前端：
1. 用新 token 覆盖旧 token。
2. 调 `GET /api/roles-permissions/user/menus`（此时 JWT 已是新租户上下文 → 返回新租户的菜单）。
3. 触发 `tenantChanged` 事件，重置路由守卫、清空旧租户列表页缓存。
4. `navigate` 到当前页或默认首页。

**实时性保证**：权限完全由 JWT 内的 `tenant_id`+`role` 驱动，切换即重签，所有后续请求自动落到新上下文，无任何旧状态残留。

### 5.3 权限解析算法
给定请求 `(user U, tenant T, role R)`：

```
function resolveMenus(U, T, R):
    roleRef = U.tenant_role_id @ T  ??  systemRole(U.role @ T)
    # 1) 角色基线
    base = tenant_role_menus[roleRef]  ??  role_menu_permissions[R]   # 租户表为空→回退系统模板
    # 2) 用户级覆盖
    overridden = base.merge(user_menu_permissions[U,T])              # 显式 0/1 覆盖
    # 3) 模块门控（粗粒度）
    gated = overridden.filter(m => tenant_module_menus[T, moduleOf(m)].enabled)
    # 4) 特权角色
    if R == super_admin: return ALL
    if R == system_admin: return gated - SENSITIVE_SYSTEM_MENUS       # /tenants,/database-connection,/cloud-sync
    return gated

function resolveDataScope(U, T, R):
    return user_data_scopes[U,T] ?? tenant_role_data_scopes[roleRef] ?? role_data_scopes[R,T] ?? 'self'

function resolveActions(U, T, R):
    allowed = tenant_role_permissions[roleRef] ∪ user_permissions[U,T]
    denied  = user_permission_denies[U,T]
    return allowed − denied
```

### 5.4 租户管理员自配角色权限 —— 满足需求④
租户管理员（`system_admin`，租户级）能力（全部受 `req.user.tenant_id === resource.tenant_id` 约束）：

| 操作 | 端点 | 说明 |
|---|---|---|
| 查本租户角色 | `GET /api/tenant/roles` | 仅本租户 |
| 新建角色 | `POST /api/tenant/roles` | 克隆系统模板或自定义 |
| 编辑角色 | `PUT /api/tenant/roles/:id` | 系统模板克隆体可改，原系统模板只读 |
| 配角色菜单 | `PUT /api/tenant/roles/:id/menus` | 批量设置 menu_key 可见性 |
| 配角色数据范围 | `PUT /api/tenant/roles/:id/data-scope` | all/department/self/custom |
| 配角色操作权限 | `PUT /api/tenant/roles/:id/permissions` | 从 tenant_permissions 目录选授 |
| 给用户分角色 | `POST /api/users/:id/roles` | 写 user_tenant_roles（限本租户） |
| 查操作权限目录 | `GET /api/tenant/permissions` | 全局目录，租户选用 |

**隔离守护**：每个端点中间件 `ensureTenantScope` 校验 `resource.tenant_id === req.user.tenant_id`；`super_admin` 跨租户豁免。系统原模板角色（`is_system_template=0 且来自 roles 表`）标记只读，禁止删除/改 code。

## 6. JWT 与会话设计

```json
// Access Token Claims
{
  "user_id": 220,
  "username": "test_engineer",
  "tenant_id": 2,                 // 当前租户（切换即变）
  "role": "maintenance_engineer", // 当前租户下的角色 code
  "tenant_role_id": 15,           // 若为租户自定义角色
  "role_type": "tenant",          // system | tenant
  "session_id": "uuid",           // 切换刷新，用于缓存失效
  "iat": ..., "exp": ...          // 建议 access 2h, refresh 7d
}
```
- 切换租户 = 重签 access+refresh，`session_id` 变更 → 服务端权限缓存按 `session_id` 失效。
- 临时登录 token（多租户选择阶段）无 `tenant_id`，仅允许调 `switch-tenant`，5 分钟过期。
- `super_admin` token `tenant_id=null`、`role_type=system`，可访问 `/tenants` 跨租户管理。

## 7. 安全与隔离

- **租户行级过滤**：所有业务查询走 `addTenantFilter(req)`（已存在），以 JWT `tenant_id` 为准。
- **越权校验**：切换目标租户必须存在 `user_tenant_roles` 活跃记录；否则 403。
- **租户管理员边界**：`ensureTenantScope` 中间件 + 资源归属校验，防跨租户改角色/用户。
- **系统模板保护**：`roles`/`role_menu_permissions` 对租户只读；租户改的是 `tenant_*` 副本。
- **审计**：`audit_logs` 记录 (user, tenant, action, resource)，切换租户单独记 `tenant_switch` 动作。
- **防重放**：`session_id` + 短时 access token；切换/登出使旧 session 失效。

## 8. 缓存与性能

- 服务端缓存：权限解析结果按 `(user_id, tenant_id, role, session_id)` 缓存，TTL 5 分钟；角色/菜单配置变更时按租户批量失效。
- 前端缓存：菜单列表按 `tenant_id` 缓存；切换时清旧租户缓存。
- `tenant_module_menus`（3329 行）按租户加载一次进内存，请求时 O(1) 查表。
- 热点：`/user/menus` 在切换后必调一次，确保结果可缓存（ETag/短 TTL）。

## 9. 从现有系统迁移路径（4 阶段，低风险渐进）

**阶段 1：补齐切换能力（满足②③，用现有全局角色）**
- 新增 `POST /api/users/switch-tenant`，重签 JWT。
- 改前端 `EnterpriseSelect`/`handleTenantChange`：选择后调 switch-tenant 换 token，再拉菜单。
- 多租户登录返回临时 token + 企业列表。
- 验收：同一用户在 A 租户是工程师、B 租户是资产管理员，切换后菜单实时变化。
- 风险：低，不改角色模型。

**阶段 2：引入 per-tenant 角色表（满足④骨架，回退兼容）**
- 建 `tenant_roles`/`tenant_role_menus`/`tenant_role_data_scopes`/`tenant_role_permissions`/`tenant_permissions`。
- 为每个现有租户克隆 12 个系统角色进 `tenant_roles`（`is_system_template=1`），菜单从 `role_menu_permissions` 复制进 `tenant_role_menus`。
- `user_tenant_roles` 加 `tenant_role_id` 列；迁移脚本把现有映射指向克隆出的 `tenant_role_id`。
- 权限解析引擎：优先读 `tenant_role_*`，为空回退 `role_menu_permissions`。
- 验收：行为与阶段 1 完全一致（克隆等价），但数据已落到租户表。

**阶段 3：租户管理员配置 UI**
- 实现第 5.4 节端点 + 前端「角色管理 / 菜单授权 / 数据范围 / 操作权限 / 用户分角色」页面。
- `ensureTenantScope` 守护。
- 验收：租户 A 管理员新建角色并授权菜单，不影响租户 B；租户 B 看不到 A 的角色。

**阶段 4：清理**
- `role_menu_permissions` 降级为纯系统模板，运行时不再直接读取（除回退）。
- 统一两套菜单树（`routes/menus.js` 的 `ALL_MENUS` 并入 `menu_definitions`）。
- 废弃 `user_tenant_roles.role` 字符串列（全部迁到 `tenant_role_id`）。

## 10. 验收标准（对应四条需求）

| 需求 | 验收用例 |
|---|---|
| ① | 用户 U 关联租户 A(工程师)+B(资产管理员)，登录返回两个企业 |
| ② | U 选 A → 菜单=工程师集；选 B → 菜单=资产管理员集；DB 无串号 |
| ③ | U 在 A 内打开工单页 → 切到 B → 工单页按 B 权限/数据范围实时变化；旧 token 失效 |
| ④ | 租户 A 管理员新建角色 R_A 并授权菜单 M；租户 B 用户看不到 R_A、M 不受 A 配置影响 |
| 隔离 | 租户 A 管理员尝试改 B 的角色 → 403；跨租户查资产 → 空集 |
| 实时 | 切换后 `/user/menus` 立即返回新租户菜单，无 5s 内残留 |

## 11. 风险与对策

| 风险 | 对策 |
|---|---|
| 切换重签 token 期间有在途请求 | 前端切换时设全局 loading 拦截；后端按 `session_id` 拒绝旧 session 写操作 |
| 克隆 12 角色×21 租户=252 行配置漂移 | 阶段 2 用脚本幂等克隆；阶段 3 后系统模板变更不再自动同步到租户（由租户管理员主动） |
| 缓存与实时性冲突 | `session_id` 维度缓存；角色/菜单变更发事件按租户失效 |
| 现有 `user_tenant_roles.role` 字符串与 `tenant_role_id` 并存 | 解析层统一 `effective_role` 函数；阶段 4 清字符串列 |
| super_admin 跨租户与租户隔离冲突 | `role_type=system` 走特权分支，绕过 `ensureTenantScope`，单独审计 |

### 12.1 与原方案的两处差异（已确认更优）
1. **不新增 `POST /switch-tenant` 端点**。探查发现 `middleware/auth.js` 的 `authenticate` 早已支持 `X-Tenant-ID` 请求头——每次请求按该头重查 `user_tenant_roles` 校验并覆盖 JWT 中的 role。因此"切换"只需前端在请求头带上目标租户，无需重签 JWT，更简单且实时。需求②③ 改为**纯前端接线**实现。
2. **角色模型采用"按 (tenant_id, role) 覆盖标准角色"，而非设计稿的"自定义租户角色码 + `tenant_role_id`"**。原因：`auth.js`/`roles-permissions.js` 深度耦合 12 个 `role_code` 字符串，自定义角色码侵入性大、风险高；覆盖模型零侵入、向后兼容（租户表有值用租户值，无值自动回退全局模板）。
