# 资产调配模块差错排查与完善报告

> 排查范围：`backend/routes/transfer.js`、`backend/routes/assets/asset.transfer.js`、
> `backend/services/transfer.service.js`、`backend/services/transfer-approval-service.js`、
> `backend/scripts/create-asset-transfer-requests-table.js` 及前端 `TransferForm / TransferList / TransferRequestList`。
> 结论：模块存在**双数据源不一致、删除错表、租户隔离泄漏、状态机断裂**四类核心缺陷，
> 已全部修复并统一到单一数据源 `asset_transfer_requests`。

---

## 一、模块架构概览（排查后厘清）

| 入口 | 路由 | 操作表 | 前端调用 | 现状 |
|------|------|--------|----------|------|
| 调配申请表单 | `POST /api/assets/:id/transfer-apply` | `asset_transfer_requests` | `TransferForm` | **在用（主流程）** |
| 申请列表/处理 | `GET/POST /api/assets/transfer-requests[/:id/approve]` | `asset_transfer_requests` | `TransferRequestList` | **在用（主流程）** |
| 调配记录列表 | `GET /api/transfer` | `asset_transfer_requests` | `TransferList` | 在用（已标记 deprecated） |
| 调配记录删除 | `DELETE /api/transfer/:id` | ❌误删 `transfer_records` | `TransferList` | **在用但有 BUG** |
| 调配创建/审批/完成 | `POST/PUT /api/transfer[/:id/approve,/complete]` | `transfer_records` | 无（AI/遗留） | deprecated，逻辑与主流程冲突 |
| AI 助手服务 | `TransferService.*` | `asset_transfer_requests` | AI 助手 | 逻辑错误 |

**核心结论**：业务真实运行的数据源是 `asset_transfer_requests`；`transfer_records` 是历史遗留表（已有 `drop-transfer-records-table.js` 计划下线），其状态枚举为中文、与主表英文枚举并存，是混乱根源。本次修复**统一以 `asset_transfer_requests` 为唯一数据源**，抽出共享核心 `transfer-approval-service.js`。

## 二、排查发现的问题清单

| # | 位置 | 严重度 | 类型 | 根因 | 影响 |
|---|------|--------|------|------|------|
| 1 | `routes/transfer.js` `DELETE /:id` | 🔴 严重 | 数据一致性 | 先 `SELECT ... FROM asset_transfer_requests`，却 `DELETE ... FROM transfer_records` | 删除落在错误表：申请删不掉/删错记录；资产状态（调配中）无法回滚 |
| 2 | `services/transfer-approval-service.js` 拒绝分支 | 🔴 严重 | 租户隔离 | 判断“其它未完成调拨”时查询 `transfer_records` **未带 `tenant_id`** | 跨租户误判，可能错误回滚/不回滚资产状态 |
| 3 | `services/transfer.service.js` `createTransfer` | 🔴 严重 | 逻辑错误 | 直接 INSERT，未校验资产存在/状态，**且未把资产置为“调配中”** | 经此创建的申请，审批时因资产非“调配中”而**永远无法批准** |
| 4 | `services/transfer.service.js` `completeTransfer` | 🔴 严重 | 数据不一致 | 写入 `status='completed'`，而 `ENUM` 仅 `pending/approved/rejected` | 完成调拨必抛 DB 错误，事务回滚，调拨卡死 |
| 5 | `services/transfer.service.js` `approveTransfer/rejectTransfer` | 🟠 高 | 逻辑错误 | 仅改申请状态，不更新资产部门、不要求“调配中” | 与核心逻辑不一致，资产实际部门不变化 |
| 6 | `routes/transfer.js` `POST /` | 🟠 高 | 数据一致性 | 写入 `transfer_records` 而列表读取 `asset_transfer_requests` | 经此创建的记录不出现在列表；资产被置“调配中”却无对应申请可审批 |
| 7 | 双表并存 | 🟠 高 | 架构 | `transfer_records`(中文状态) 与 `asset_transfer_requests`(英文状态) 两套枚举 | 状态模型混乱，跨表判断易出错 |
| 8 | `routes/transfer.js` 列表/详情 `CASE` | 🟡 中 | 展示 | 状态映射未覆盖 `completed`，前端却有“已完成”标签无对应状态 | 状态展示歧义 |
| 9 | `scripts/create-asset-transfer-requests-table.js` | 🔴 严重(阻塞) | 脚本错误 | 列名写成 `asset.code`（非法），且**缺 `tenant_id` 列** | 脚本本身建表失败；线上表缺 `tenant_id` 依赖迁移补偿，隔离脆弱 |
| 10 | 审批/删除异常分支 | 🟡 中 | 异常流程 | 缺乏资产不存在/状态异常的细粒度提示与一致性回滚 | 异常时提示不友好、回滚不完整 |

## 三、统一修复方案

### 3.1 状态机（唯一权威）
```
在用  --申请(校验通过,加行锁)-->  调配中  --审批通过(实际调拨)-->  在用(目标科室)
                                          └--驳回/删除(无其它待审批)-->  在用(原科室)
```
- **不存在独立的 `completed` 状态**：审批通过即代表调拨完成（资产部门与状态已更新）。
- 申请状态枚举固定：`pending / approved / rejected`。

### 3.2 共享核心 `transfer-approval-service.js`
抽出统一函数，所有调用方复用，消除重复与不一致：
- `findTransferRequestById(connection, id, tenantId)`
- `getAssetForTransfer(connection, tenantId, assetIdentifier)` —— 兼容 `asset_code`/数值 `id`，`FOR UPDATE` 行锁
- `hasOtherPendingTransfer(connection, assetCode, tenantId, excludeId)` —— **带 `tenant_id`**（修复 #2）
- `createTransferRequest(...)` —— 校验资产存在/在用/去重，置“调配中”后入库（修复 #3、#6）
- `approveTransferRequest(...)` —— 要求“调配中”，更新资产部门+状态+变更日志，置 `approved`（修复 #1 的连锁、#5）
- `rejectTransferRequest(...)` / `deleteTransferRequest(...)` —— 按需回滚资产状态，删除**正确表**（修复 #1）

所有函数**不在内部 commit/rollback**，由调用方统一事务管理；业务错误返回 `{success, message, code}`，便于路由映射准确 HTTP 码与回滚。

### 3.3 调用方改造
| 文件 | 改造点 |
|------|--------|
| `routes/assets/asset.transfer.js` | `transfer-apply`、`approve` 改为调用共享核心，删除重复校验与租户泄漏查询 |
| `routes/transfer.js` | `DELETE` 改用 `deleteTransferRequest`（删对表）；`POST/approve/complete` 统一到 `asset_transfer_requests`；`complete` 幂等（已批准直接成功）；状态映射补全 |
| `services/transfer.service.js` | `create/approve/complete/reject` 全部委托共享核心；`completeTransfer` 去掉非法 `completed`，改为“审批即完成”幂等语义 |
| `scripts/create-asset-transfer-requests-table.js` | `asset.code`→`asset_code`，补 `tenant_id`、索引，ENUM 修正 |

## 四、边界条件与异常流程处理

1. **并发申请去重**：申请时对目标资产加 `FOR UPDATE` 行锁，并校验“无其它 pending 申请”，杜绝 TOCTOU 重复提交。
2. **资产状态机校验**：仅“在用”可发起；审批要求“调配中”；拒绝/删除时仅当资产仍为“调配中”且**无其它待审批申请**才回滚为“在用”。
3. **已审批申请删除不回滚**：调拨已生效，资产保留在目标科室（修复了误把已完成资产弹回原科室的风险）。
4. **幂等**：`complete` 对已 `approved` 申请直接返回成功；`approve/reject` 对已处理申请返回明确提示，不重复执行。
5. **租户隔离**：所有读写均带 `tenant_id`，拒绝/删除的“其它待审批”判断同样带 `tenant_id`（修复 #2）。
6. **遗留表兼容**：删除/清理 `transfer_records` 时捕获 `ER_NO_SUCH_TABLE`，表已下线也不阻塞主流程。
7. **错误提示**：业务校验返回中文化、可定位的 message 与对应 HTTP 码（400/404/500），便于前端 `message.error` 直接展示。

## 六、验证与回归建议

**接口/单元测试用例**
1. 申请：资产“在用” → 成功且资产变“调配中”；资产“调配中”/“报废”等 → 400。
2. 并发申请同一资产 → 仅一条 pending。
3. 审批 pending 申请 → 资产部门=目标科室、状态=“在用”、申请=approved。
4. 审批非“调配中”资产 → 400（资产状态机断裂保护）。
5. 驳回/删除 pending（无其它 pending）→ 资产回滚“在用”；存在其它 pending → 不回滚。
6. 删除 approved 申请 → 资产**不**回滚（调拨已生效）。
7. `DELETE /api/transfer/:id` → 确认 `asset_transfer_requests` 行消失、资产状态正确（回归 #1）。
8. 跨租户：A 租户操作不影响 B 租户资产（回归 #2）。

**数据一致性自检 SQL**
```sql
-- 应无“调配中”却无对应 pending 申请的孤儿资产
SELECT a.asset_code, a.status
FROM assets a
WHERE a.status = '调配中'
  AND NOT EXISTS (
    SELECT 1 FROM asset_transfer_requests t
    WHERE t.asset_code = a.asset_code AND t.tenant_id = a.tenant_id AND t.status = 'pending'
  );

-- 应无 status 非法值的申请（枚举约束保障）
SELECT * FROM asset_transfer_requests WHERE status NOT IN ('pending','approved','rejected');
```

## 七、遗留与后续建议

1. **`transfer_records` 表**：已有下线脚本，建议后续版本将 `routes/transfer.js` 的 admin 直调审计改为仅写 `asset_change_logs`，并彻底移除 `transfer_records` 相关代码与建表/删除脚本。
2. **前端 `TransferList` 删除**：当前依赖 `DELETE /api/transfer/:id`（deprecated 路由），本次已修复；建议中长期将其迁移到 `/api/assets` 下的统一调配接口，关闭 deprecated 路由。
3. **`completeTransfer` 语义**：Service 层“完成”现等价于“审批通过”。若业务确实需要“申请→审批→出库完成”三段式，应在 `asset_transfer_requests` 增加 `completed` 状态并调整核心，但当前与线上运行模型保持一致优先。
