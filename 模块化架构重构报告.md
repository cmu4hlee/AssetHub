# AssetHub 模块化架构重构报告

**日期**: 2025-05-03  
**版本**: v1.0.0  
**状态**: 实施完成

---

## 一、重构目标

1. **模块独立性**: 每个模块能独立运行、维护和扩展
2. **松耦合架构**: 消除模块间不必要的直接依赖
3. **接口规范化**: 为每个模块设计独立的接口规范
4. **测试独立性**: 建立模块独立的测试体系
5. **配置独立化**: 各模块独立配置运行参数
6. **错误隔离**: 单个模块异常不影响整体系统

```

│   └── requests.service.js → notification.service.js
├── quality-control (质量控制模块)
```

### 2.2 发现的问题

| 问题 | 严重程度 | 影响 |
|------|---------|------|
| 模块间直接依赖其他模块的服务 | 🔴 高 | 无法独立部署和测试 |
| 没有统一的错误处理机制 | 🟡 中 | 错误信息不一致，难以排查 |
| 缺少模块配置管理 | 🟡 中 | 配置散落各处，难以维护 |
| 没有模块注册机制 | 🟡 中 | 模块加载顺序不确定 |
| 缺乏模块测试框架 | 🟢 低 | 无法验证模块独立性 |

## 三、架构设计方案

### 3.1 模块接口规范

创建文件: [module-interfaces.js](file:///Volumes/移动硬盘（500）/AssetHub/backend/architecture/module-interfaces.js)

定义了以下标准接口：

| 接口 | 说明 |
|------|------|
| `StandardServiceInterface` | 标准服务接口（findById, findAll, create, update, delete） |
| `ModuleConfigInterface` | 模块配置接口 |
| `ModuleEventInterface` | 模块事件接口（emit, on, off） |
| `ModuleLifecycleInterface` | 模块生命周期接口（initialize, start, stop, destroy） |
| `RepositoryInterface` | 数据访问接口（Repository 模式） |
| `ModuleException` | 标准异常类 |

### 3.2 模块注册中心

创建文件: [module-registry.js](file:///Volumes/移动硬盘（500）/AssetHub/backend/config/module-registry.js)

功能：
- 模块注册与管理
- 依赖关系验证
- 拓扑排序初始化
- 事件总线通信
- 模块状态监控

### 3.3 错误处理机制

创建文件: [module-error-handler.js](file:///Volumes/移动硬盘（500）/AssetHub/backend/middleware/module-error-handler.js)

功能：
- 统一异常处理中间件
- 异步路由包装器 (`asyncHandler`)
- 模块隔离中间件
- 安全执行函数 (`safeExecute`)
- 带重试的安全执行 (`safeExecuteWithRetry`)

### 3.4 配置管理器

创建文件: [module-config-manager.js](file:///Volumes/移动硬盘（500）/AssetHub/backend/config/module-config-manager.js)

功能：
- 模块配置注册
- 多级配置覆盖（默认 < 环境变量 < 配置文件）
- 配置验证
- 配置监听（热更新）
- 嵌套配置支持

### 3.5 测试框架

创建文件: [module-test-framework.js](file:///Volumes/移动硬盘（500）/AssetHub/backend/testing/module-test-framework.js)

功能：
- 模块测试上下文
- Mock 依赖支持
- 接口契约测试
- 模块隔离测试
- 测试报告生成

## 四、模块间通信方式

### 4.1 事件驱动通信

```
模块 A ──emit──→ 事件总线 ──on──→ 模块 B
```

**优点**:
- 模块间无直接引用
- 支持一对多、多对多通信
- 异步非阻塞

**示例**:
```javascript
// 模块 A 发布事件
moduleRegistry.emitEvent('asset', 'created', { assetCode: 'A001' });

// 模块 B 监听事件
moduleRegistry.onEvent('asset', 'created', (payload) => {
  // 处理资产创建事件
});
```

### 4.2 服务接口调用（通过注册中心）

```
模块 A ──moduleRegistry.get('asset')──→ 资产服务
```

**优点**:
- 明确的依赖关系
- 支持 Mock 测试
- 统一管理

## 五、错误处理机制

### 5.1 标准异常类型

| 异常代码 | HTTP状态 | 说明 |
|---------|---------|------|
| `VALIDATION_ERROR` | 400 | 参数验证失败 |
| `NOT_FOUND` | 404 | 记录不存在 |
| `FORBIDDEN` | 403 | 权限不足 |
| `CONFLICT` | 409 | 业务规则冲突 |
| `DEPENDENCY_ERROR` | 503 | 外部依赖失败 |
| `INTERNAL_ERROR` | 500 | 系统内部错误 |

### 5.2 异常处理流程

```
路由处理函数
    ↓ (抛出异常)
asyncHandler 包装器
    ↓ (捕获异常)
moduleErrorHandler 中间件
    ↓ (格式化响应)
HTTP 响应
```

### 5.3 隔离保护

```javascript
// 模块隔离中间件确保单个模块异常不影响其他模块
router.use(moduleIsolationMiddleware('quality-control'));
```

## 八、使用示例

### 8.1 注册模块

```javascript
const moduleRegistry = require('./config/module-registry');
const QualityControlService = require('./services/quality-control-service');

moduleRegistry.register('quality-control', new QualityControlService(), {
  config: {
    warningDays: 30,
    autoRemind: true,
  },
  dependencies: ['asset'],
  enabled: true,
});
```

### 8.2 使用模块

```javascript
const { asyncHandler, createModuleError } = require('./middleware/module-error-handler');
const moduleRegistry = require('./config/module-registry');

router.post('/quality-control', asyncHandler(async (req, res) => {
  const qcService = moduleRegistry.get('quality-control');
  if (!qcService) {
    throw createModuleError('DEPENDENCY_ERROR', '质量控制模块未注册');
  }

  const result = await qcService.create(req.body, req.tenantId, req.user?.username);
  res.json({ success: true, data: result });
}));
```

### 8.3 模块间事件通信

```javascript
// 资产模块发布事件
moduleRegistry.emitEvent('asset', 'status-changed', {
  assetCode: 'A001',
  oldStatus: '在用',
  newStatus: '维修',
});

// 质量控制模块监听
moduleRegistry.onEvent('asset', 'status-changed', (payload) => {
  if (payload.newStatus === '维修') {
    // 触发质控检查
  }
});
```
