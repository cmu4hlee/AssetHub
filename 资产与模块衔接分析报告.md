# 资产（assets）与各功能模块衔接分析报告

> 数据源：本地 zcgl 数据库实际表结构 + 后端代码静态扫描
> 库内总表数：279 ；引用 assets 的表：73 个 ；总引用字段：85 个

---

## 一、整体概览

### 1.1 引用规模

| 维度 | 数值 | 备注 |
|---|---|---|
| 引用资产的主表/子表 | 73 | 横跨 30+ 业务模块 |
| `asset_id INT` 类型引用 | 33 处 | 14 个表的 `asset_id` 字段 |
| `asset_code VARCHAR` 类型引用 | 52 处 | 59+ 个表的 `asset_code` 字段 |
| 实际有外键约束的 | **仅 5 个** | `asset_ai_analysis_logs`、`asset_shares`、`idle_assets`、`location_alert_rules`、`location_alerts` |
| 同一表冗余存 `asset_id` + `asset_code` | 7 张 | 见下表 |

### 1.2 同时存在 asset_id 和 asset_code 的表（冗余风险）

| 表名 | 现状 | 实际使用 |
|---|---|---|
| `adverse_reaction_records` | 两列都在 | 仅用 `asset_code` |
| `adverse_reaction_records_active` | 两列都在 | 仅用 `asset_code` |
| `idle_assets` | 两列都在 | 视场景混用 |
| `inspection_records` | 两列都在 | JOIN 用 `asset_id` |
| `location_alert_rules` | 两列都在 | 业务层按场景选 |
| `location_alerts` | 两列都在 | 外键用 `asset_id` |
| `maintenance_plans` | 两列都在 | JOIN 用 `asset_code` |

**风险**：写多份就要多份同步，任意一边更新另一边不跟就会数据漂移。资产表里 `asset_code` 理论上可被修改（维护状态在 `asset_change_logs` 里有专门的记录机制），但子表的 `asset_code` 是静态字符串，**改资产编码即等于让所有子表的关联关系失效**。

## 二、问题分级清单

### 🔴 P0：影响数据一致性的硬伤

#### 1. 73 张引用表里 68 张没有外键约束

只有 5 张表对 `assets` 建了 FK：
```
asset_ai_analysis_logs.asset_id      → assets.id
asset_shares.asset_id                → assets.id
idle_assets.asset_id                 → assets.id
location_alert_rules.asset_id        → assets.id
location_alerts.asset_id             → assets.id
```
其余 68 张表（包括 `maintenance_requests`、`work_orders`、`warranty_info` 等核心业务表）用 `asset_code VARCHAR` 裸引用，没有任何数据库层面的关联保护。结果：
- `assets` 表里 `asset_code` 唯一性由 `UNIQUE KEY uk_tenant_asset_code (tenant_id, asset_code)` 兜底，但子表拿不到这个保护。
- 删除/重编码/跨租户搬运资产时无法级联，孤儿数据靠人工/业务代码兜底。

**代表表（高风险、NOT NULL、无 FK）：**
```
maintenance_requests.asset_code
work_orders.asset_code
warranty_info.asset_code / warranty_contracts / warranty_history / warranty_reminders
risk_assessments.asset_id
uptime_statistics.asset_id
asset_change_logs.asset_code
inventory_details.asset_code / inventory_details_active / inventory_discrepancies
asset_images.asset_code
asset_locations.asset_code / asset_location_history
asset_scrapping_records.asset_code
asset_transfers.asset_code
```

#### 2. 软删除未级联（`is_deleted = 1` 后子表全部失联）

`assets` 是**软删除**（`is_deleted` + `deleted_at`），但代码里各模块 JOIN `assets` 时**多数不写 `a.is_deleted = 0`**：

| 模块 / 文件 | JOIN 现状 | 影响 |
|---|---|---|
| `maintenance/requests.service.js:141,165` | 裸 JOIN | 维修申请里会显示已删资产的工单 |
| `maintenance/workorders.service.js:323` | 裸 JOIN | 工单详情/列表出现"幽灵资产" |
| `maintenance/costs.service.js:84,122,201` | 裸 JOIN | 维修成本统计混入已删资产 |
| `inspection-management/inspection.service.js:13` | 裸 JOIN | 巡检任务/问题列表里 asset_name = NULL |
| `inspection-management/inspection-extended.service.js` | 裸 JOIN | 巡检路线点显示空 |
| `asset-risk-management/risk-assessment.service.js` | 裸 JOIN | 风险评估失去资产名 |
| `asset-usage-management/asset-usage.service.js` | 裸 JOIN | 使用记录统计错位 |
| `inventory-management/inventory.service.js` | 裸 JOIN | 盘点时仍能查到已删资产 |
| `depreciation-management/depreciation.service.js:304` | **连过滤都没有** | **继续给已报废资产算折旧** |
| `uptime-management/uptime.service.js` | 裸 JOIN | 开机率统计错 |
| `special-equipment-management` | 裸 JOIN | 特种设备与已删资产挂钩 |
| `tendering-management` | 裸 JOIN | 招标与已删资产挂钩 |
| `safety-inspection-management` | 裸 JOIN | 安全检查失联 |

只有 `maintenance/plans.service.js:371` 里硬过滤了 `a.is_deleted = 0`（仅这一处），其他都没做。

**业务后果**：
- 报表面板"资产总数"和"在用"数对不上。
- 工单/巡检里显示"未关联资产"或重复记录。
- 折旧多算（钱数错）。
- 维修费用按已删资产重复扣减（合规风险）。

#### 3. `asset_code` 同时是业务主键 + 外键，可被改

`assets` 表上 `asset_code` 是 UNIQUE（`uk_tenant_asset_code`），**不是主键**但所有子表都拿它当外键用。`asset.service.js:343` 提供的 `updateAsset` 也没在白名单里把 `asset_code` 排除（看代码没改写它），但 `asset_change_logs` 写变更日志时**主键是 `asset_code`**：

```js
// asset.service.js:479
(asset_code, tenant_id, changed_by, field_name, old_value, new_value, changed_at)
```

> 这里有歧义：按业务常见做法，**资产编码是不应该让用户改的**。如果项目允许改，就要写一个"级联同步"机制把所有子表里的 `asset_code` 一起更新；目前没有这层机制 → 改 `asset_code` = 静默破坏 68 张表的关联。

### 🟠 P1：模块衔接的合理性问题

#### 4. 同一资产被多种方式引用

| 引用方式 | 出现的模块 |
|---|---|
| `asset_id` 外键（推荐） | 巡检、风险、特种设备、计量、IoT 区域/告警、开机率、采购验收/发票 |
| `asset_code` 字符串（历史遗留） | 维修、保养、质检、保修、盘点、资产变更日志、资产位置、资产使用 |
| 双向冗余（id + code） | 闲置资产、巡检、维修计划、位置告警/告警规则、不良事件 |

**问题**：
- 前端要兼容两种入参（`asset_id` vs `asset_code`），`asset.service.js:323` 专门写了 `resolveAssetIdentifier` 兼容，给维护增加心智负担。
- 跨模块联查时要先确认这张表用的是什么字段，无法统一 `JOIN assets a ON a.id = x.asset_id`。
- 报表/分析需要做多种关联策略，性能和正确性都难保证。

#### 5. 模块之间的引用路径不一致

| 源表 | 当前关系 | 业务上应该 |
|---|---|---|
| `maintenance_requests` → 资产 | 软引用 `asset_code` | 强引用 `asset_id` |
| `work_orders` → 资产 | 软引用 `asset_code` | 强引用 `asset_id` |
| `warranty_info/contracts/history` | 软引用 `asset_code` | 强引用 `asset_id` |
| `inventory_details` | 软引用 `asset_code` | 强引用 `asset_id`（盘点是按资产实体进行的） |
| `asset_change_logs` | 软引用 `asset_code` | 这个**可以**用 code（审计日志快照），但建议同时记 `asset_id` |
| `asset_locations` / `asset_location_history` | 软引用 `asset_code` | 强引用 `asset_id`（位置是按资产 ID 绑的） |
| `iot_*_ts` 时间序列表 | 软引用 `asset_code` | 这类时序数据**不应该**每条都 JOIN assets，应该按 `asset_code` + 分区查，但要在写入时校验资产存在 |

#### 6. 旧表 vs 新表的"影子表"问题

以下表出现"active"后缀的影子表，且都跟资产相关：

```
asset_acceptance_records          ↔ asset_acceptance_records_active
asset_scrapping_records           ↔ asset_scrapping_records_active
adverse_reaction_records          ↔ adverse_reaction_records_active
inventory_details                 ↔ inventory_details_active
maintenance_usage_records         ↔ maintenance_usage_triggered
metrology_records                 ↔ metrology_records_active
```

影子表没有 FK 到 assets，且和主表的字段结构高度相似。读代码看不出这俩的差别（"_active" 名字暗示是分区/归档后的活跃表，但 FK 引用关系是断裂的）。

#### 7. 资产删除/报废路径没有做"联动校验"

`asset.service.js:561 deleteAsset` 软删一行就完事，**不检查任何引用**：
- 维修工单/申请状态还是 `待审批/维修中`？
- 保修还在生效期？
- 盘点单还没结案？
- 巡检任务还排着？
- 折旧还在计提？

正确做法：删除/报废前做"反向校验"或在子表加 `ON DELETE RESTRICT`（因为软删走不到这层，至少在 service 层拦截）。

### 🟡 P2：细节建议（不修不致命，但建议优化）

#### 8. `asset.service.js` 的 `resolveAssetIdentifier` 不该作为通用方法下沉到业务层

```js
// asset.service.js:323
resolveAssetIdentifier(idOrCode) {
  // ...
  if (/^\d+$/.test(raw) && raw.length <= 12) {
    return { column: 'id', value: ... };
  }
  return { column: 'asset_code', value: raw };
}
```

`/^\d+$/.test(raw) && raw.length <= 12` 这条规则会让 `asset_code = "000123"` 被错误地当成 id。
建议：
- 把"主键 vs 业务键"显式交给前端传 `?type=id` 或 `?type=code`。
- 或者在表里加 `asset_code` 用前缀（例如 `A-`）让字符串辨识度更高，避免和 id 撞车。

#### 9. `assets` 表的索引重复严重

`SHOW INDEX FROM assets` 出来 **27 条**索引，重复模式：
```
idx_code:                  asset_code
idx_assets_asset_code:     asset_code
idx_assets_tenant_code:    tenant_id, asset_code
idx_assets_tenant_asset_code: tenant_id, asset_code
```

优化建议：合并为单条 `UNIQUE KEY uk_tenant_asset_code (tenant_id, asset_code)`（已经存在），删掉单独的 `idx_code` / `idx_assets_asset_code`。其余重复索引清理。

#### 10. `depreciation-management` 模块对资产的语义错了

```js
// depreciation.service.js:304
const baseFrom = 'FROM assets a';
```

`queryAssets` 用 `a.purchase_date DESC, a.id DESC` 排序，且**不带 `is_deleted` 过滤**——这意味着折旧模块会拿软删资产、报废资产来计提。需要加 `a.is_deleted = 0 AND a.status != '报废'` 之类的业务过滤。

## 三、推荐整改路线

按"投入产出比"分三步走：

### 第一步（1-2 天）：补齐软删除过滤和删除前校验
- 给所有 `LEFT JOIN assets a` 统一加 `AND a.is_deleted = 0`。
- 改写 `deleteAsset`，加引用校验（不允许有未结维修、未到期的保修、未完成的盘点就软删）。
- 折旧模块加 `a.is_deleted = 0 AND a.status != '报废'` 过滤。

### 第二步（3-5 天）：收口两种引用方式
- 业务上确定一个权威主键（建议 **`asset_id`**，因为它不会变）。
- 把 `maintenance_requests` / `work_orders` / `warranty_*` / `inventory_*` / `asset_locations` / `asset_change_logs` 等关键表的 `asset_code` 字段保留为"快照"（用于显示和审计），**新增一列 `asset_id` 作为主关联**。
- 数据迁移：写脚本回填 `asset_id`，加 `UNIQUE` 复合索引。
- 评估是否要落 FK（折中：开发库落 FK，线上库只建索引）。

### 第三步（持续）：清理影子表和重复索引
- 决定 `*_active` 表是不是分区/归档策略，如果是，写到 model 里；如果不是，合并掉。
- 清理 `assets` 表 27 条索引里重复的 8-10 条。
- 对所有 `asset_code` 字符串引用做兜底校验：写入前必须能查到对应资产（service 层 `SELECT id FROM assets WHERE asset_code = ? AND tenant_id = ? AND is_deleted = 0`）。

## 六、结论

资产是这套系统的中心实体，但**目前的设计是"逻辑关联 + 字符串主引用"**，没有充分利用 FK 也没有统一软删除协议。

- **数据完整性**：依赖代码层兜底，68 张表无任何数据库层面的保护。
- **查询正确性**：软删除未级联，导致报表/统计容易"算多了"或"显空了"。
- **演进成本**：两种引用方式并存，`asset_code` 既是业务键又被当外键，未来要改资产编码或做资产合并很痛苦。

建议按"先补软删除过滤和删除校验 → 再做引用收口 → 最后清索引和影子表"的三步路线整改。第一步能在不破坏现有接口的前提下立刻止血，建议优先做。

> 完整数据：见本报告第三节 73 张引用表清单和第五节代码位置，可直接据此排期。
